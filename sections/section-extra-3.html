23 DBA Isbirligi ve Performans Analizi

23.1 DBA'larin Degeri

Neden DBA ile Calismaliyiz?

Thorben'in hikayesi: Gelistirici ekibi saatler boyunca bir sorguyu optimize edemedi. DBA birkac saat icinde sorguyu 1 saniyenin altina dusurdu. Sebep basit: DBA'lar tamamen farkli bir beceri setine sahiptir. SQL optimizasyonu, execution plan analizi ve index stratejisi konusunda uzmanlardir.

Ideal Durum: Ekipte bir DBA olmasi ve development surecine aktif olarak katilmasi. DBA sadece production sorunlarinda degil, gelistirme asamasinda da degerlidir.

23.2 Haftalik DBA Review Sureci

Pratik Oneri: Haftalik (veya iki haftalik) DBA Review

Ozel bir test run yapilir ve DBA asagidakileri kontrol eder:

  1. En yavas sorgular: Execution time'i yuksek sorgulari tespit eder
  2. En sik calisan sorgular: Gereksiz tekrar eden sorgulari bulur
  3. Sonuclari standup meeting'e getirir ve ekiple paylesir

DBA Perspektifinden N+1 Tespiti: "Yeni bir sorgu buldum ve 3500 kere calistiriyorsunuz!" — Bu tipik bir N+1 problemidir. Gelistirici kodda farketmeyebilir, ancak DBA veritabani tarafinda bunu hemen gorur. DBA'nin bu tur bulgulari standup'a getirmesi, performans sorunlarini erken yakalamak icin kritiktir.

23.3 EXPLAIN ANALYZE ile Sorgu Analizi

EXPLAIN ANALYZE Nedir?

PostgreSQL'de bir sorgunun nasil calistigini gosteren komuttur. Sorguyu gercekten calistirir ve her adimin ne kadar surdigini, kac satir isledigini detayli olarak raporlar.

-- Basit bir sorgunun execution plan'ini gormek
EXPLAIN ANALYZE
SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id
WHERE o.status = 'PENDING'
ORDER BY o.created_at DESC;

EXPLAIN ANALYZE Ciktisini Okuma

-- Ornek EXPLAIN ANALYZE ciktisi:
-- Sort  (cost=156.34..158.67 rows=932 width=128) (actual time=12.456..12.890 rows=950 loops=1)
--   Sort Key: o.created_at DESC
--   Sort Method: quicksort  Memory: 142kB
--   ->  Hash Join  (cost=25.10..112.45 rows=932 width=128) (actual time=1.234..8.567 rows=950 loops=1)
--         Hash Cond: (oi.order_id = o.id)
--         ->  Seq Scan on order_items oi  (cost=0.00..72.50 rows=3250 width=64) (actual time=0.012..2.345 rows=3250 loops=1)
--         ->  Hash  (cost=20.35..20.35 rows=380 width=64) (actual time=0.987..0.987 rows=380 loops=1)
--               Buckets: 1024  Batches: 1  Memory Usage: 45kB
--               ->  Seq Scan on orders o  (cost=0.00..20.35 rows=380 width=64) (actual time=0.008..0.567 rows=380 loops=1)
--                     Filter: (status = 'PENDING')
--                     Rows Removed by Filter: 620
-- Planning Time: 0.234 ms
-- Execution Time: 13.012 ms

-- DIKKAT EDILECEKLER:
-- 1. "Seq Scan" goruyorsaniz: Tablo full taranmis, index eksik olabilir
-- 2. "actual time" vs "cost": Gercek sure ile tahmini maliyeti karsilastirin
-- 3. "Rows Removed by Filter": Cok yuksekse, index eklenmeli
-- 4. "loops": 1'den buyukse, ic ice dongu var (N+1 belirtisi olabilir)

23.4 Index Stratejisi

Index Temelleri

Index, veritabaninda arama islemlerini hizlandiran bir veri yapidir (kitabin indeksi gibi). Ancak her index INSERT ve UPDATE islemlerini yavaslatir cunku index de guncellenmek zorundadir.

Low-Hanging Fruit: Foreign Key Kolonlarinda Index

Foreign key kolonlarina index eklemek en kolay ve en etkili performans iyilestirmesidir. JOIN islemlerinde buyuk fark yaratir.

@Entity
@Table(name = "order_items", indexes = {
    @Index(name = "idx_order_item_order_id", columnList = "order_id"),
    @Index(name = "idx_order_item_product_id", columnList = "product_id")
})
public class OrderItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id")
    private Product product;

    private int quantity;
    private BigDecimal unitPrice;
    // ...
}

Cok Fazla Index de Sorun! Her index INSERT ve UPDATE islemlerini yavaslatir. Bir tabloda 15-20 index varsa, yazma islemleri ciddi sekilde etkilenir. Index eklerken okuma/yazma dengesini dusunun.

Composite Index (Birlesik Index)

// Composite Index: Birden fazla kolon icin tek index
// Kullanim senaryosu: WHERE status = ? AND created_at > ? seklinde
// sik kullanilan sorgular varsa
@Entity
@Table(name = "orders", indexes = {
    @Index(name = "idx_order_status_date", columnList = "status, created_at"),
    @Index(name = "idx_order_customer_id", columnList = "customer_id")
})
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String status;

    @Column(name = "created_at")
    private LocalDateTime createdAt;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;
    // ...
}

Composite Index Sirasi Onemlidir: columnList = "status, created_at" seklinde tanimlanan index, WHERE status = ? sorgularinda da calisir ama WHERE created_at > ? sorgularinda calismaz. Sol-dan-saga (leftmost prefix) kurali gecerlidir.

23.5 Database Vendor Araclari

Veritabani Arac Kullanim
PostgreSQL pg_stat_statements En sik calisan ve en yavas sorgulari listeler
MySQL slow_query_log Belirli esik degerinden yavas calisan sorgulari loglar
Oracle AWR Report Performans istatistiklerini detayli raporlar 
-- PostgreSQL: En yavas 10 sorguyu bul
SELECT query,
       calls,
       mean_exec_time::numeric(10,2) AS avg_ms,
       total_exec_time::numeric(10,2) AS total_ms
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- PostgreSQL: En sik calisan 10 sorguyu bul
SELECT query,
       calls,
       mean_exec_time::numeric(10,2) AS avg_ms
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;
-- MySQL: Slow query log'u aktif et
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 1 saniyeden yavas sorgulari logla
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

Bilgi Kontrolu

1. DBA'nin haftalik review'da kontrol etmesi gereken iki temel metrik nedir?

✓ Dogru! DBA en yavas sorgulari (performance bottleneck) ve en sik calisan sorgulari (N+1 gibi tekrar problemleri) kontrol etmelidir.
✗ Yanlis. Dogru cevap B: DBA haftalik review'da en yavas sorgulari ve en sik calisan sorgulari kontrol eder. Bu iki metrik performans sorunlarini ve N+1 gibi tekrar problemlerini ortaya cikarir.

2. EXPLAIN ANALYZE ciktisinda "Seq Scan" gormek ne anlama gelir?

✓ Dogru! Seq Scan (Sequential Scan), tablonun bastan sona tarandigini gosterir. Buyuk tablolarda bu cok yavas olabilir ve genellikle uygun bir index eklenmesi gerektigini isaret eder.
✗ Yanlis. Dogru cevap A: Seq Scan, tablonun tamaminin tarandigini gosterir. Bu buyuk tablolarda performans sorunu yaratir ve genellikle eksik bir index'e isaret eder.

3. Foreign key kolonlarina index eklemek neden "low-hanging fruit" olarak adlandirilir?

✓ Dogru! FK kolonlarindaki index, JOIN islemlerini onemli olcude hizlandirir ve eklenmesi cok kolaydir. Minimum efor, maksimum kazanc saglar.
✗ Yanlis. Dogru cevap C: Foreign key index'leri minimum eforla eklenir ve JOIN islemlerinde buyuk performans kazanci saglar. Bu yuzden "kolay meyve" (low-hanging fruit) olarak adlandirilir.

24 Production Benzeri Ortamda Test

24.1 Problem: Local vs Production Farki

Thorben'in Uyarisi: "Local in-memory database ile test etmekten hoslanmiyorum." Sebep: Development ortaminda birkac duzine kayit varken, production'da milyonlarca kayit bulunur. Performans sorunlari development'ta gozukmez, ancak production'da patlak verir.

H2 ile Test Etmenin Riskleri

H2 in-memory database ile test etmek yaniltici sonuclar verir:

  • SQL uyumlulugu sinirli: H2'nin PostgreSQL/MySQL modu tam uyumlu degildir
  • Dialect farkliliklari: Bazi SQL fonksiyonlari H2'de farkli calisir veya desteklenmez
  • Veri hacmi: 10 satirda calisan sorgu, 10 milyon satirda timeout alabilir
  • Index etkileri: Kucuk veri setlerinde index olsa da olmasa da fark etmez

24.2 Best Practice Hierarchy

Test Ortami Hiyerarsisi (En iyiden en kotusune)

  1. Ayni DB motoru ile test: Testcontainers ile PostgreSQL/MySQL container'i calistir
  2. Anonimlestirilmis production verisi: Gercek veri hacimleriyle test pipeline'i
  3. Dedicated performans test suite'i: Buyuk veri setleri ile ozel performans testleri

24.3 Testcontainers ile Entegrasyon Testi

Testcontainers Nedir?

Testcontainers, testler sirasinda Docker container'lari baslatip yoneten bir Java kutuphanesidir. Her test icin temiz bir veritabani instance'i saglar ve test bittiginde otomatik olarak temizler.

Maven Bagimliligi

<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>testcontainers</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>

Spring Boot Test Kurulumu

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
@Testcontainers
class OrderRepositoryIntegrationTest {

    // Her test sinifi icin bir PostgreSQL container'i baslatilir
    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine")
        .withDatabaseName("testdb")
        .withUsername("test")
        .withPassword("test");

    // Spring Boot'un datasource ayarlarini container'a yonlendir
    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
        registry.add("spring.datasource.driver-class-name", () -> "org.postgresql.Driver");
        registry.add("spring.jpa.hibernate.ddl-auto", () -> "create-drop");
    }

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private ProductRepository productRepository;

    @Test
    void shouldFindOrdersWithItems() {
        // Given: Test verisi olustur
        Product product = new Product();
        product.setName("Test Urun");
        product.setPrice(new BigDecimal("99.90"));
        productRepository.save(product);

        Order order = new Order();
        order.setStatus("PENDING");
        order.setCreatedAt(LocalDateTime.now());
        orderRepository.save(order);

        // When: Gercek PostgreSQL uzerinde sorgu calistir
        List<Order> orders = orderRepository.findByStatus("PENDING");

        // Then: Sonuclari dogrula
        assertThat(orders).hasSize(1);
        assertThat(orders.get(0).getStatus()).isEqualTo("PENDING");
    }

    @Test
    void shouldHandleBulkOperationsEfficiently() {
        // Buyuk veri seti ile performans testi
        for (int i = 0; i < 1000; i++) {
            Product p = new Product();
            p.setName("Product " + i);
            p.setPrice(BigDecimal.valueOf(i * 10));
            productRepository.save(p);
        }

        // Gercek DB uzerinde sorgu performansini test et
        long start = System.currentTimeMillis();
        List<Product> products = productRepository.findAll();
        long duration = System.currentTimeMillis() - start;

        assertThat(products).hasSize(1000);
        assertThat(duration).isLessThan(5000); // 5 saniyeden kisa olmali
    }
}

Ipucu: @Container anotasyonu ile isaretlenen container, test sinifi baslamadan once otomatik olarak baslatilir ve testler bittikten sonra durdurulur. @DynamicPropertySource sayesinde Spring Boot'un veritabani ayarlari, container'in dinamik port ve URL bilgilerine yonlendirilir.

application-test.yml Konfigurasyonu

# src/test/resources/application-test.yml
spring:
  jpa:
    hibernate:
      ddl-auto: create-drop  # Test icin her seferinde yeniden olustur
    show-sql: false           # Test ciktisini temiz tut
    properties:
      hibernate:
        format_sql: true
        generate_statistics: true  # Hibernate istatistiklerini ac
        session.events.log.LOG_QUERIES_SLOWER_THAN_MS: 500  # 500ms'den yavas sorgulari logla

  # Testcontainers kullanildiginda datasource ayarlari
  # DynamicPropertySource ile override edilir
  datasource:
    driver-class-name: org.postgresql.Driver

logging:
  level:
    org.hibernate.stat: DEBUG        # Hibernate istatistikleri
    org.hibernate.SQL: WARN          # SQL loglarini sadece uyari seviyesinde goster

24.4 Anonimlestirilmis Production Verisi

Neden Anonimlestirilmis Veri?

Gercek performans testleri icin gercek veri hacmi gerekir. Ancak production verisinde hassas bilgiler (kredi karti, email, isim) vardir. Cozum: Hassas verileri cikarip veri hacmini korumak.

Anonimlestirilecek Veriler

  • Kredi karti numaralari: **** **** **** 1234 gibi maskelenir
  • Email adresleri: user_12345@anonymous.test ile degistirilir
  • Isim ve soyisim: Rastgele degerlerle degistirilir
  • Telefon numaralari: Maskelenir veya rastgele uretilir

Korunan: Veri hacmi (milyon satir), iliski yapilari, veri dagilimi — performans testleri icin bunlar kritiktir.

-- Ornek: Production verisini anonimlestirilmis test verisine donusturme
-- 1. Production dump'ini al
-- 2. Hassas verileri anonimlestr

UPDATE customers SET
    first_name = 'User_' || id,
    last_name = 'Test_' || id,
    email = 'user_' || id || '@anonymous.test',
    phone = '+90555' || LPAD(id::text, 7, '0'),
    credit_card = NULL
WHERE id > 0;

-- 3. Veri hacmi korunur: 5 milyon musteri, 20 milyon siparis aynen kalir
-- 4. Bu veri seti ile performans testleri gercekci sonuclar verir

24.5 SQL Monitoring: show-sql Yerine Dogru Araclar

YANLIS -- spring.jpa.show-sql=true

show-sql sadece SQL'i konsola yazdirir. Parametreleri gostermez, istatistik toplamaz, yavasliga sebep olabilir.

# Bunu kullanmayin (ozellikle test pipeline'da)
spring:
  jpa:
    show-sql: true  # Parametreleri gostermez, performansi etkiler

DOGRU -- datasource-proxy veya p6spy kullanin

Bu araclar SQL sorgularini parametreleriyle birlikte loglar, calisma surelerini olcer ve sorgu sayisini takip eder.

<!-- datasource-proxy Maven bagimliligi -->
<dependency>
    <groupId>net.ttddyy</groupId>
    <artifactId>datasource-proxy</artifactId>
    <version>1.10</version>
</dependency>

<!-- veya p6spy -->
<dependency>
    <groupId>p6spy</groupId>
    <artifactId>p6spy</artifactId>
    <version>3.9.1</version>
</dependency>
// datasource-proxy ile SQL monitoring konfigurasyonu
@Configuration
public class DataSourceProxyConfig {

    @Bean
    public DataSource dataSource(DataSource originalDataSource) {
        return ProxyDataSourceBuilder.create(originalDataSource)
            .name("SQL-Log")
            .logQueryBySlf4j(SLF4JLogLevel.INFO)  // SLF4J ile logla
            .multiline()                           // SQL'i okunakli formatla
            .countQuery()                          // Sorgu sayisini takip et
            .build();
    }
}

datasource-proxy vs p6spy: Her ikisi de SQL sorgularini parametreleriyle birlikte loglar. datasource-proxy daha hafiftir ve Spring Boot ile entegrasyonu kolaydir. p6spy ise daha eski ve yaygin kullanilan bir alternatiftir. Test pipeline'larinda bu araclardan birini kullanarak N+1 gibi sorunlari otomatik olarak tespit edebilirsiniz.

Thorben'in Son Uyarisi: "Production'da calismiyorsa sikayet etmeyin, daha once hic kontrol etmediyseniz." Gercek veritabani, gercek veri hacmi ve gercek sorgularla test etmek her gelistiricinin sorumlulugudur. H2 ile yazilan testler gecse bile, production'da sorun yasanabilir.

Bilgi Kontrolu

1. H2 in-memory database ile test etmenin en buyuk riski nedir?

✓ Dogru! H2'nin SQL uyumlulugu sinirlidir ve birkac duzine kayit ile calistirilan testler, milyonlarca kayit iceren production'daki performans sorunlarini gosteremez.
✗ Yanlis. Dogru cevap C: H2'nin SQL dialect'i gercek veritabanlariyla tam uyumlu degildir ve kucuk veri setleri performans sorunlarini gizler. Production'da milyonlarca kayit oldugundan, H2 testleri yaniltici olabilir.

2. Testcontainers'da @DynamicPropertySource ne ise yarar?

✓ Dogru! @DynamicPropertySource, container her seferinde farkli bir portta basladigi icin Spring Boot'un datasource konfigurasyonunu calisma zamaninda dinamik olarak ayarlar.
✗ Yanlis. Dogru cevap B: @DynamicPropertySource, Testcontainers'in dinamik olarak atadigi port ve URL bilgilerini Spring Boot'un datasource ayarlarina baglar. Container her seferinde farkli bir portta baslayabilecegi icin bu dinamik yapilandirma gereklidir.

3. Production verisini test icin kullanirken neden anonimlestirilmesi gerekir?

✓ Dogru! Hassas veriler (kredi karti, email, isim) gizlilik ve guvenlik acisindan cikarilmali veya maskelenmeli, ancak veri hacmi (milyon satir) ve veri dagilimi korunmalidir ki performans testleri gercekci sonuc versin.
✗ Yanlis. Dogru cevap A: Anonimlestirilmenin amaci hassas verileri (kredi karti, email, isim) korumaktir. Veri hacmi ve dagilimi korunarak performans testlerinin gercekci sonuc vermesi saglanir.
