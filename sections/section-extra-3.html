23 DBA İşbirliği ve Performans Analizi

23.1 DBA'ların Değeri

Neden DBA ile Çalışmalıyız?

Thorben'in hikayesi: Geliştirici ekibi saatler boyunca bir sorguyu optimize edemedi. DBA birkaç saat içinde sorguyu 1 saniyenin altına düşürdü. Sebep basit: DBA'lar tamamen farklı bir beceri setine sahiptir. SQL optimizasyonu, execution plan analizi ve index stratejisi konusunda uzmanlardır.

İdeal Durum: Ekipte bir DBA olması ve development sürecine aktif olarak katılması. DBA sadece production sorunlarında değil, geliştirme aşamasında da değerlidir.

23.2 Haftalık DBA Review Süreci

Pratik Öneri: Haftalık (veya iki haftalık) DBA Review

Özel bir test run yapılır ve DBA aşağıdakileri kontrol eder:

  1. En yavaş sorgular: Execution time'i yüksek sorguları tespit eder
  2. En sık çalışan sorgular: Gereksiz tekrar eden sorguları bulur
  3. Sonuçları standup meeting'e getirir ve ekiple paylaşır

DBA Perspektifinden N+1 Tespiti: "Yeni bir sorgu buldum ve 3500 kere çalıştırıyorsunuz!" — Bu tipik bir N+1 problemidir. Geliştirici kodda farketmeyebilir, ancak DBA veritabanı tarafında bunu hemen görür. DBA'nın bu tür bulguları standup'a getirmesi, performans sorunlarını erken yakalamak için kritiktir.

23.3 EXPLAIN ANALYZE ile Sorgu Analizi

EXPLAIN ANALYZE Nedir?

PostgreSQL'de bir sorgunun nasıl çalıştığını gösteren komuttur. Sorguyu gerçekten çalıştırır ve her adımın ne kadar sürdüğini, kaç satır işlediğini detaylı olarak raporlar.

-- Basit bir sorgunun execution plan'ını görmek
EXPLAIN ANALYZE
SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id
WHERE o.status = 'PENDING'
ORDER BY o.created_at DESC;

EXPLAIN ANALYZE Çıktısını Okuma

-- Örnek EXPLAIN ANALYZE çıktısı:
-- Sort  (cost=156.34..158.67 rows=932 width=128) (actual time=12.456..12.890 rows=950 loops=1)
--   Sort Key: o.created_at DESC
--   Sort Method: quicksort  Memory: 142kB
--   ->  Hash Join  (cost=25.10..112.45 rows=932 width=128) (actual time=1.234..8.567 rows=950 loops=1)
--         Hash Cond: (oi.order_id = o.id)
--         ->  Seq Scan on order_items oi  (cost=0.00..72.50 rows=3250 width=64) (actual time=0.012..2.345 rows=3250 loops=1)
--         ->  Hash  (cost=20.35..20.35 rows=380 width=64) (actual time=0.987..0.987 rows=380 loops=1)
--               Buckets: 1024  Batches: 1  Memory Usage: 45kB
--               ->  Seq Scan on orders o  (cost=0.00..20.35 rows=380 width=64) (actual time=0.008..0.567 rows=380 loops=1)
--                     Filter: (status = 'PENDING')
--                     Rows Removed by Filter: 620
-- Planning Time: 0.234 ms
-- Execution Time: 13.012 ms

-- DİKKAT EDILECEKLER:
-- 1. "Seq Scan" görüyorsanız: Tablo full taranmış, index eksik olabilir
-- 2. "actual time" vs "cost": Gerçek süre ile tahmini maliyeti karşılaştırın
-- 3. "Rows Removed by Filter": Çok yüksekse, index eklenmeli
-- 4. "loops": 1'den büyükse, iç içe döngü var (N+1 belirtisi olabilir)

23.4 Index Stratejisi

Index Temelleri

Index, veritabanında arama işlemlerini hızlandıran bir veri yapıdır (kitabın indeksi gibi). Ancak her index INSERT ve UPDATE işlemlerini yavaşlatır çünkü index de güncellenmek zorundadır.

Low-Hanging Fruit: Foreign Key Kolonlarında Index

Foreign key kolonlarına index eklemek en kolay ve en etkili performans iyileştirmesidir. JOIN işlemlerinde büyük fark yaratır.

@Entity
@Table(name = "order_items", indexes = {
    @Index(name = "idx_order_item_order_id", columnList = "order_id"),
    @Index(name = "idx_order_item_product_id", columnList = "product_id")
})
public class OrderItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id")
    private Product product;

    private int quantity;
    private BigDecimal unitPrice;
    // ...
}

Çok Fazla Index de Sorun! Her index INSERT ve UPDATE işlemlerini yavaşlatır. Bir tabloda 15-20 index varsa, yazma işlemleri ciddi şekilde etkilenir. Index eklerken okuma/yazma dengesini düşünün.

Composite Index (Birleşik Index)

// Composite Index: Birden fazla kolon için tek index
// Kullanım senaryosu: WHERE status = ? AND created_at > ? şeklinde
// sık kullanılan sorgular varsa
@Entity
@Table(name = "orders", indexes = {
    @Index(name = "idx_order_status_date", columnList = "status, created_at"),
    @Index(name = "idx_order_customer_id", columnList = "customer_id")
})
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String status;

    @Column(name = "created_at")
    private LocalDateTime createdAt;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;
    // ...
}

Composite Index Sırası Önemlidir: columnList = "status, created_at" şeklinde tanımlanan index, WHERE status = ? sorgularında da çalışır ama WHERE created_at > ? sorgularında çalışmaz. Soldan-sağa (leftmost prefix) kuralı geçerlidir.

23.5 Database Vendor Araçları

Veritabanı Araç Kullanım
PostgreSQL pg_stat_statements En sık çalışan ve en yavaş sorguları listeler
MySQL slow_query_log Belirli eşik değerinden yavaş çalışan sorguları loglar
Oracle AWR Report Performans istatistiklerini detaylı raporlar 
-- PostgreSQL: En yavaş 10 sorguyu bul
SELECT query,
       calls,
       mean_exec_time::numeric(10,2) AS avg_ms,
       total_exec_time::numeric(10,2) AS total_ms
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- PostgreSQL: En sık çalışan 10 sorguyu bul
SELECT query,
       calls,
       mean_exec_time::numeric(10,2) AS avg_ms
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;
-- MySQL: Slow query log'u aktif et
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 1 saniyeden yavaş sorguları logla
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

Bilgi Kontrolü

1. DBA'nın haftalık review'da kontrol etmesi gereken iki temel metrik nedir?

✓ Doğru! DBA en yavaş sorguları (performance bottleneck) ve en sık çalışan sorguları (N+1 gibi tekrar problemleri) kontrol etmelidir.
✗ Yanlış. Doğru cevap B: DBA haftalık review'da en yavaş sorguları ve en sık çalışan sorguları kontrol eder. Bu iki metrik performans sorunlarını ve N+1 gibi tekrar problemlerini ortaya çıkarır.

2. EXPLAIN ANALYZE çıktısında "Seq Scan" görmek ne anlama gelir?

✓ Doğru! Seq Scan (Sequential Scan), tablonun baştan sona tarandığını gösterir. Büyük tablolarda bu çok yavaş olabilir ve genellikle uygun bir index eklenmesi gerektiğini işaret eder.
✗ Yanlış. Doğru cevap A: Seq Scan, tablonun tamamının tarandığını gösterir. Bu büyük tablolarda performans sorunu yaratır ve genellikle eksik bir index'e işaret eder.

3. Foreign key kolonlarına index eklemek neden "low-hanging fruit" olarak adlandırılır?

✓ Doğru! FK kolonlarındaki index, JOIN işlemlerini önemli ölçüde hızlandırır ve eklenmesi çok kolaydır. Minimum efor, maksimum kazanç sağlar.
✗ Yanlış. Doğru cevap C: Foreign key index'leri minimum eforla eklenir ve JOIN işlemlerinde büyük performans kazancı sağlar. Bu yüzden "kolay meyve" (low-hanging fruit) olarak adlandırılır.

24 Production Benzeri Ortamda Test

24.1 Problem: Local vs Production Farkı

Thorben'in Uyarısı: "Local in-memory database ile test etmekten hoşlanmıyorum." Sebep: Development ortamında birkaç düzine kayıt varken, production'da milyonlarca kayıt bulunur. Performans sorunları development'ta gözükmez, ancak production'da patlak verir.

H2 ile Test Etmenin Riskleri

H2 in-memory database ile test etmek yanıltıcı sonuçlar verir:

  • SQL uyumluluğu sınırlı: H2'nin PostgreSQL/MySQL modu tam uyumlu değildir
  • Dialect farklılıkları: Bazı SQL fonksiyonları H2'de farklı çalışır veya desteklenmez
  • Veri hacmi: 10 satırda çalışan sorgu, 10 milyon satırda timeout alabilir
  • Index etkileri: Küçük veri setlerinde index olsa da olmasa da fark etmez

24.2 Best Practice Hierarchy

Test Ortamı Hiyerarşisi (En iyiden en kötüsüne)

  1. Aynı DB motoru ile test: Testcontainers ile PostgreSQL/MySQL container'ı çalıştır
  2. Anonimleştirilmiş production verisi: Gerçek veri hacimleriyle test pipeline'ı
  3. Dedicated performans test suite'i: Büyük veri setleri ile özel performans testleri

24.3 Testcontainers ile Entegrasyon Testi

Testcontainers Nedir?

Testcontainers, testler sırasında Docker container'ları başlatıp yöneten bir Java kütüphanesidir. Her test için temiz bir veritabanı instance'ı sağlar ve test bittiğinde otomatik olarak temizler.

Maven Bağımlılığı

<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>testcontainers</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>1.19.7</version>
    <scope>test</scope>
</dependency>

Spring Boot Test Kurulumu

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
@Testcontainers
class OrderRepositoryIntegrationTest {

    // Her test sınıfı için bir PostgreSQL container'ı başlatılır
    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine")
        .withDatabaseName("testdb")
        .withUsername("test")
        .withPassword("test");

    // Spring Boot'un datasource ayarlarını container'a yönlendir
    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
        registry.add("spring.datasource.driver-class-name", () -> "org.postgresql.Driver");
        registry.add("spring.jpa.hibernate.ddl-auto", () -> "create-drop");
    }

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private ProductRepository productRepository;

    @Test
    void shouldFindOrdersWithItems() {
        // Given: Test verisi oluştur
        Product product = new Product();
        product.setName("Test Ürün");
        product.setPrice(new BigDecimal("99.90"));
        productRepository.save(product);

        Order order = new Order();
        order.setStatus("PENDING");
        order.setCreatedAt(LocalDateTime.now());
        orderRepository.save(order);

        // When: Gerçek PostgreSQL üzerinde sorgu çalıştır
        List<Order> orders = orderRepository.findByStatus("PENDING");

        // Then: Sonuçları doğrula
        assertThat(orders).hasSize(1);
        assertThat(orders.get(0).getStatus()).isEqualTo("PENDING");
    }

    @Test
    void shouldHandleBulkOperationsEfficiently() {
        // Büyük veri seti ile performans testi
        for (int i = 0; i < 1000; i++) {
            Product p = new Product();
            p.setName("Product " + i);
            p.setPrice(BigDecimal.valueOf(i * 10));
            productRepository.save(p);
        }

        // Gerçek DB üzerinde sorgu performansını test et
        long start = System.currentTimeMillis();
        List<Product> products = productRepository.findAll();
        long duration = System.currentTimeMillis() - start;

        assertThat(products).hasSize(1000);
        assertThat(duration).isLessThan(5000); // 5 saniyeden kısa olmalı
    }
}

İpucu: @Container anotasyonu ile işaretlenen container, test sınıfı başlamadan önce otomatik olarak başlatılır ve testler bittikten sonra durdurulur. @DynamicPropertySource sayesinde Spring Boot'un veritabanı ayarları, container'ın dinamik port ve URL bilgilerine yönlendirilir.

application-test.yml Konfigürasyonu

# src/test/resources/application-test.yml
spring:
  jpa:
    hibernate:
      ddl-auto: create-drop  # Test için her seferinde yeniden oluştur
    show-sql: false           # Test çıktısını temiz tut
    properties:
      hibernate:
        format_sql: true
        generate_statistics: true  # Hibernate istatistiklerini aç
        session.events.log.LOG_QUERIES_SLOWER_THAN_MS: 500  # 500ms'den yavaş sorguları logla

  # Testcontainers kullanıldığında datasource ayarları
  # DynamicPropertySource ile override edilir
  datasource:
    driver-class-name: org.postgresql.Driver

logging:
  level:
    org.hibernate.stat: DEBUG        # Hibernate istatistikleri
    org.hibernate.SQL: WARN          # SQL loglarını sadece uyarı seviyesinde göster

24.4 Anonimleştirilmiş Production Verisi

Neden Anonimleştirilmiş Veri?

Gerçek performans testleri için gerçek veri hacmi gerekir. Ancak production verisinde hassas bilgiler (kredi kartı, email, isim) vardır. Çözüm: Hassas verileri çıkarıp veri hacmini korumak.

Anonimleştirilecek Veriler

  • Kredi kartı numaraları: **** **** **** 1234 gibi maskelenir
  • Email adresleri: user_12345@anonymous.test ile değiştirilir
  • İsim ve soyisim: Rastgele değerlerle değiştirilir
  • Telefon numaraları: Maskelenir veya rastgele üretilir

Korunan: Veri hacmi (milyon satır), ilişki yapıları, veri dağılımı — performans testleri için bunlar kritiktir.

-- Örnek: Production verisini anonimleştirilmiş test verisine dönüştürme
-- 1. Production dump'ını al
-- 2. Hassas verileri anonimleştir

UPDATE customers SET
    first_name = 'User_' || id,
    last_name = 'Test_' || id,
    email = 'user_' || id || '@anonymous.test',
    phone = '+90555' || LPAD(id::text, 7, '0'),
    credit_card = NULL
WHERE id > 0;

-- 3. Veri hacmi korunur: 5 milyon müşteri, 20 milyon sipariş aynen kalır
-- 4. Bu veri seti ile performans testleri gerçekçi sonuçlar verir

24.5 SQL Monitoring: show-sql Yerine Doğru Araçlar

YANLIŞ -- spring.jpa.show-sql=true

show-sql sadece SQL'i konsola yazdırır. Parametreleri göstermez, istatistik toplamaz, yavaşlığa sebep olabilir.

# Bunu kullanmayın (özellikle test pipeline'da)
spring:
  jpa:
    show-sql: true  # Parametreleri göstermez, performansı etkiler

DOĞRU -- datasource-proxy veya p6spy kullanın

Bu araçlar SQL sorgularını parametreleriyle birlikte loglar, çalışma sürelerini ölçer ve sorgu sayısını takip eder.

<!-- datasource-proxy Maven bağımlılığı -->
<dependency>
    <groupId>net.ttddyy</groupId>
    <artifactId>datasource-proxy</artifactId>
    <version>1.10</version>
</dependency>

<!-- veya p6spy -->
<dependency>
    <groupId>p6spy</groupId>
    <artifactId>p6spy</artifactId>
    <version>3.9.1</version>
</dependency>
// datasource-proxy ile SQL monitoring konfigürasyonu
@Configuration
public class DataSourceProxyConfig {

    @Bean
    public DataSource dataSource(DataSource originalDataSource) {
        return ProxyDataSourceBuilder.create(originalDataSource)
            .name("SQL-Log")
            .logQueryBySlf4j(SLF4JLogLevel.INFO)  // SLF4J ile logla
            .multiline()                           // SQL'i okunaklı formatla
            .countQuery()                          // Sorgu sayısını takip et
            .build();
    }
}

datasource-proxy vs p6spy: Her ikisi de SQL sorgularını parametreleriyle birlikte loglar. datasource-proxy daha hafiftir ve Spring Boot ile entegrasyonu kolaydır. p6spy ise daha eski ve yaygın kullanılan bir alternatiftir. Test pipeline'larında bu araçlardan birini kullanarak N+1 gibi sorunları otomatik olarak tespit edebilirsiniz.

Thorben'in Son Uyarısı: "Production'da çalışmıyorsa şikayet etmeyin, daha önce hiç kontrol etmediyseniz." Gerçek veritabanı, gerçek veri hacmi ve gerçek sorgularla test etmek her geliştiricinin sorumluluğudur. H2 ile yazılan testler geçse bile, production'da sorun yaşanabilir.

Bilgi Kontrolü

1. H2 in-memory database ile test etmenin en büyük riski nedir?

✓ Doğru! H2'nin SQL uyumluluğu sınırlıdır ve birkaç düzine kayıt ile çalıştırılan testler, milyonlarca kayıt içeren production'daki performans sorunlarını gösteremez.
✗ Yanlış. Doğru cevap C: H2'nin SQL dialect'i gerçek veritabanlarıyla tam uyumlu değildir ve küçük veri setleri performans sorunlarını gizler. Production'da milyonlarca kayıt olduğundan, H2 testleri yanıltıcı olabilir.

2. Testcontainers'da @DynamicPropertySource ne işe yarar?

✓ Doğru! @DynamicPropertySource, container her seferinde farklı bir portta başladığı için Spring Boot'un datasource konfigürasyonunu çalışma zamanında dinamik olarak ayarlar.
✗ Yanlış. Doğru cevap B: @DynamicPropertySource, Testcontainers'in dinamik olarak atadığı port ve URL bilgilerini Spring Boot'un datasource ayarlarına bağlar. Container her seferinde farklı bir portta başlayabileceği için bu dinamik yapılandırma gereklidir.

3. Production verisini test için kullanırken neden anonimleştirilmesi gerekir?

✓ Doğru! Hassas veriler (kredi kartı, email, isim) gizlilik ve güvenlik açısından çıkarılmalı veya maskelenmeli, ancak veri hacmi (milyon satır) ve veri dağılımı korunmalıdır ki performans testleri gerçekçi sonuç versin.
✗ Yanlış. Doğru cevap A: Anonimleştirilmenin amacı hassas verileri (kredi kartı, email, isim) korumaktır. Veri hacmi ve dağılımı korunarak performans testlerinin gerçekçi sonuç vermesi sağlanır.
