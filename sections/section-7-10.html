7 Spring Data JPA — Üst Düzey Soyutlama

Spring Data JPA, Hibernate ve JPA üzerine kurulu bir soyutlama katmanıdır. Repository arayüzleri tanımlayarak, hiç SQL yazmadan CRUD işlemleri yapmanızı sağlar.

Repository Pattern

Spring Data JPA, repository arayüzleri üzerinden veri erişimini soyutlar. JpaRepository arayüzünü extend ederek, hiçbir implementasyon yazmadan temel CRUD işlemlerini ve daha fazlasını elde edersiniz. Spring Data, metot adlarından otomatik olarak SQL üretir.

7.1 Temel Repository Tanımı

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.jpa.repository.Modifying;
import org.springframework.data.repository.query.Param;

public interface ProductRepository extends JpaRepository<Product, Long> {

    // ─── Turetilmis Sorgular (Derived Queries) ───
    // Spring Data metot adindan SQL uretir

    // SELECT * FROM products WHERE category = ?
    List<Product> findByCategory(String category);

    // SELECT * FROM products WHERE category = ? AND status = ?
    // DIKKAT: En fazla 2 parametre! Daha fazlasi okunaksiz olur.
    List<Product> findByCategoryAndStatus(String category, ProductStatus status);

    // SELECT * FROM products WHERE price BETWEEN ? AND ?
    List<Product> findByPriceBetween(BigDecimal min, BigDecimal max);

    // SELECT * FROM products WHERE name LIKE '%keyword%'
    List<Product> findByNameContainingIgnoreCase(String keyword);

    // ─── JPQL ile Ozel Sorgular ───
    @Query("SELECT p FROM Product p WHERE p.price > :minPrice ORDER BY p.price DESC")
    List<Product> findExpensiveProducts(@Param("minPrice") BigDecimal minPrice);

    // ─── DTO Projection (Sadece gerekli alanlari cek) ───
    @Query("SELECT new com.example.dto.ProductSummary(p.name, p.price) " +
           "FROM Product p WHERE p.category = :category")
    List<ProductSummary> findSummaryByCategory(@Param("category") String category);

    // ─── Native SQL (Veri tabanina ozel fonksiyonlar gerektiginde) ───
    @Query(value = "SELECT * FROM products WHERE is_active = true " +
                   "ORDER BY created_at DESC LIMIT 10",
           nativeQuery = true)
    List<Product> findTop10ActiveProducts();

    // ─── Toplu Guncelleme (Bulk Update) ───
    @Modifying
    @Query("UPDATE Product p SET p.price = p.price * :multiplier WHERE p.category = :cat")
    int bulkUpdatePrices(@Param("cat") String category, @Param("multiplier") double multiplier);
}
İpucu: Türetilmiş sorgularda (Derived Queries) en fazla 2 parametre kullanın. Daha fazlası metot adını okunaksız hale getirir. Karmaşık sorgular için @Query anotasyonunu tercih edin.

7.2 Servis Katmanı Kullanımı

@Service
public class ProductService {

    @Autowired
    private ProductRepository productRepository;

    // Yeni urun olustur
    @Transactional
    public Product createProduct(ProductCreateDTO dto) {
        Product product = new Product();
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
        product.setCategory(dto.getCategory());
        product.setStatus(ProductStatus.ACTIVE);
        product.setCreatedAt(LocalDateTime.now());
        return productRepository.save(product);
        // Burada save() ZORUNLU cunku nesne yeni (Transient).
        // save() onu Managed yapar ve ID atar.
    }

    // Mevcut urunu guncelle
    @Transactional
    public void updateProduct(Long id, ProductUpdateDTO dto) {
        Product product = productRepository.findById(id).orElseThrow();
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
        // save() YOK! Nesne Managed, Dirty Checking yapacak.
    }

    // Silme
    @Transactional
    public void deleteProduct(Long id) {
        Product product = productRepository.findById(id).orElseThrow();
        productRepository.delete(product);
    }

    // Salt okunur sorgu (Read-Only) — Performans optimizasyonu
    @Transactional(readOnly = true)
    public List<Product> getExpensiveProducts(BigDecimal minPrice) {
        return productRepository.findExpensiveProducts(minPrice);
        // readOnly = true: Hibernate snapshot ALMAZ → Bellek tasarrufu
        // Dirty Checking YAPILMAZ → CPU tasarrufu
    }
}
Dikkat: Yeni bir nesne (Transient) oluştururken save() çağırmak zorunludur. Ancak findById() ile çekilen bir nesne zaten Managed durumundadır ve üzerindeki değişiklikler Dirty Checking ile otomatik olarak veritabanına yansır — save() gerekmez.

readOnly = true Optimizasyonu

@Transactional(readOnly = true) kullanıldığında Hibernate, entity'lerin snapshot kopyasını almaz. Bu, hem bellek tasarrufu sağlar hem de Dirty Checking yapılmadığı için CPU tasarrufu sağlar. Salt okunur sorgularda mutlaka kullanılmalıdır.

Bilgi Kontrolü

1. Spring Data JPA'da findById() ile çekilen bir entity üzerinde değişiklik yaptıktan sonra ne yapmalısınız?

✓ Doğru! Managed durumdaki entity’ler üzerindeki değişiklikler Dirty Checking mekanizması ile otomatik olarak veritabanına yansır.
✗ Yanlış. Doğru cevap B. findById() ile çekilen entity Managed durumdadır ve Dirty Checking sayesinde değişiklikler otomatik kaydedilir.

2. Türetilmiş sorgularda (Derived Queries) en fazla kaç parametre kullanılması önerilir?

✓ Doğru! En fazla 2 parametre kullanılmalı, daha fazlası metot adını okunaksız yapar. Karmaşık sorgular için @Query tercih edilmelidir.
✗ Yanlış. Doğru cevap C (2). Daha fazla parametre metot adını okunaksız hale getirir; bunun yerine @Query kullanılmalıdır.

3. @Transactional(readOnly = true) kullanmanın avantajı nedir?

✓ Doğru! readOnly modunda Hibernate, entity snapshot'ları almaz ve Dirty Checking yapmaz. Bu hem bellek hem CPU tasarrufu sağlar.
✗ Yanlış. Doğru cevap A. readOnly = true olduğunda Hibernate snapshot almaz ve Dirty Checking yapmaz, bu da bellek ve CPU tasarrufu sağlar.

8 Spring Data JDBC — Sihirsiz Yaklaşım

Spring Data JDBC, Hibernate kullanmaz. JDBC’yi doğrudan kullanır. Persistence Context, Lazy Loading, Dirty Checking gibi “sihirli” mekanizmalar yoktur. Domain Driven Design (DDD) yaklaşımına daha uygundur.

Spring Data JDBC Felsefesi

Spring Data JDBC, “açık ve net” bir yaklaşım benimser. Arka planda sihirli davranışlar yoktur. Her değişikliği açıkça save() ile kaydetmeniz gerekir. Bu, kodu daha öngörülebilir kılar ve Domain Driven Design (DDD) prensiplerine daha uygun bir yapı sunar.

8.1 Temel Farklar

Özellik Spring Data JPA Spring Data JDBC
ORM Motoru Hibernate Yok (Doğrudan JDBC)
Persistence Context Var Yok
Dirty Checking Var (Otomatik UPDATE) Yok
Lazy Loading Var Yok
save() Zorunluluğu Managed ise gereksiz HER ZAMAN zorunlu
Karmaşıklık Yüksek (sihir çok) Düşük (açık ve net)

8.2 Aggregate Root Modeli

// Spring Data JDBC — Entity yerine "Aggregate" kullanilir
import org.springframework.data.annotation.Id;
import org.springframework.data.relational.core.mapping.MappedCollection;
import org.springframework.data.relational.core.mapping.Table;

@Table("orders")
public class Order {

    @Id
    private Long id;
    private String status;
    private LocalDateTime createdAt;

    // Aggregate icindeki alt nesneler
    // Order kaydedildiginde Items da otomatik kaydedilir/silinir
    @MappedCollection(idColumn = "order_id")
    private Set<OrderItem> items = new HashSet<>();

    // Is mantigi metotlari dogrudan Entity'de
    public void addItem(Long productId, int quantity, BigDecimal price) {
        items.add(new OrderItem(productId, quantity, price));
    }

    public BigDecimal getTotalAmount() {
        return items.stream()
            .map(item -> item.getUnitPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

Aggregate Root

Spring Data JDBC, DDD'nin Aggregate Root konseptini uygular. Bir Aggregate Root (ör. Order) kaydedildiğinde, içindeki alt nesneler (ör. OrderItem) de otomatik olarak kaydedilir veya silinir. İş mantığı metotları doğrudan entity sınıfının içinde tanımlanır.

8.3 Kullanımı

@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;

    @Transactional
    public void updateOrderStatus(Long orderId, String newStatus) {
        Order order = orderRepository.findById(orderId).orElseThrow();

        order.setStatus(newStatus);

        // !! Spring Data JDBC'de save() ZORUNLU !!
        // Dirty Checking yok. Acikca kaydetmezseniz degisiklik kaybolur.
        orderRepository.save(order);
    }
}
Dikkat: Spring Data JDBC’de Dirty Checking mekanizması yoktur. Entity üzerinde yaptığınız her değişikliği save() ile açıkça kaydetmelisiniz. Aksi takdirde değişiklikleriniz kaybolur!

Bilgi Kontrolü

1. Spring Data JDBC ile Spring Data JPA arasındaki en temel fark nedir?

✓ Doğru! Spring Data JDBC, Hibernate kullanmaz ve bu yüzden Persistence Context, Dirty Checking, Lazy Loading gibi mekanizmalar bulunmaz.
✗ Yanlış. Doğru cevap C. Spring Data JDBC'de Hibernate'in “sihirli” mekanizmaları (Persistence Context, Dirty Checking, Lazy Loading) yoktur.

2. Spring Data JDBC'de bir entity’yi güncelledikten sonra ne yapmalısınız?

✓ Doğru! Spring Data JDBC'de Dirty Checking olmadığı için her değişiklikte save() çağırmak zorunludur.
✗ Yanlış. Doğru cevap A. Spring Data JDBC'de Dirty Checking yoktur, bu nedenle save() çağırmak zorunludur.

3. Spring Data JDBC hangi tasarım yaklaşımına daha uygundur?

✓ Doğru! Spring Data JDBC, Aggregate Root konseptiyle DDD yaklaşımına doğal olarak uygundur.
✗ Yanlış. Doğru cevap B. Spring Data JDBC, Aggregate Root modeli ile DDD (Domain Driven Design) yaklaşımına daha uygundur.

9 jOOQ — SQL Odaklı Tip Güvenli Sorgular

jOOQ (Java Object Oriented Querying) bir ORM değildir. SQL yazmayı sevenler için, SQL’i Java kodu içinde tip güvenli (type-safe) olarak yazmayı sağlar.

jOOQ Nedir?

jOOQ, veritabanı şemanızdan Java sınıfları üretir ve SQL sorgularını Java kodu olarak yazmanızı sağlar. Tablo yapısı değiştiğinde kodunuz derleme aşamasında hata verir (compile-time safety). Hibernate’den farklı olarak, yazdığınız SQL tam olarak ne üretiyorsa o çalışır.

9.1 Neden jOOQ?

  • Hibernate karmaşık sorgularda (çoklu JOIN, alt sorgu, pencere fonksiyonları) zorlanır
  • Hibernate’in ürettiği SQL’i kontrol edemezsiniz
  • jOOQ’da yazdığınız SQL tam olarak ne üretiyorsa o çalışır
  • Tablo yapısı değişirse, kodunuz derleme aşamasında hata verir (compile-time safety)

9.2 jOOQ ile Karmaşık Sorgu

@Service
public class ReportService {

    @Autowired
    private DSLContext dsl;  // jOOQ'nun ana nesnesi

    public List<SalesSummary> getSalesSummary(int year) {
        return dsl
            .select(
                USERS.FIRST_NAME,
                USERS.LAST_NAME,
                sum(ORDERS.TOTAL_AMOUNT).as("total_sales"),
                count(ORDERS.ID).as("order_count")
            )
            .from(USERS)
            .join(ORDERS).on(USERS.ID.eq(ORDERS.USER_ID))
            .where(
                extract(ORDERS.ORDER_DATE, DatePart.YEAR).eq(year)
                .and(ORDERS.STATUS.eq("COMPLETED"))
            )
            .groupBy(USERS.ID, USERS.FIRST_NAME, USERS.LAST_NAME)
            .having(sum(ORDERS.TOTAL_AMOUNT).gt(BigDecimal.valueOf(1000)))
            .orderBy(sum(ORDERS.TOTAL_AMOUNT).desc())
            .fetchInto(SalesSummary.class);
    }

    // Pencere fonksiyonu (Window Function) — Hibernate ile yapmak cok zor
    public List<RankedProduct> getTopProductsByCategory() {
        return dsl
            .select(
                PRODUCTS.NAME,
                PRODUCTS.CATEGORY,
                PRODUCTS.PRICE,
                rowNumber()
                    .over(partitionBy(PRODUCTS.CATEGORY).orderBy(PRODUCTS.PRICE.desc()))
                    .as("rank")
            )
            .from(PRODUCTS)
            .fetchInto(RankedProduct.class);
    }
}
İpucu: jOOQ, pencere fonksiyonları (ROW_NUMBER, RANK, DENSE_RANK), CTE’ler ve karmaşık alt sorgular gibi ileri düzey SQL özelliklerini doğal ve tip güvenli bir şekilde destekler. Bu tür sorguları Hibernate/JPQL ile yazmak çok zordur.

9.3 Ne Zaman Hibernate, Ne Zaman jOOQ?

Senaryo Araç
Basit CRUD (Oluştur, Oku, Güncelle, Sil) Hibernate / Spring Data JPA
Tekil kayıt güncelleme Hibernate (Dirty Checking)
Karmaşık raporlar, dashboard verileri jOOQ
Çoklu JOIN + alt sorgu + gruplama jOOQ
Pencere fonksiyonları (ROW_NUMBER, RANK) jOOQ
Toplu veri güncelleme (Bulk Update) jOOQ veya @Modifying JPQL
İlişki yönetimi (Cascade, Lazy Loading) Hibernate

Hibernate vs jOOQ Karşılaştırması

Hibernate / Spring Data JPA
  • Basit CRUD işlemleri için ideal
  • Entity ilişki yönetimi (Cascade, Lazy Loading)
  • Dirty Checking ile otomatik güncelleme
  • Tekil kayıt güncelleme
jOOQ
  • Karmaşık raporlar ve dashboard verileri
  • Çoklu JOIN, alt sorgu, gruplama
  • Pencere fonksiyonları (ROW_NUMBER, RANK)
  • Derleme zamanı tip güvenliği
  • Üretilen SQL üzerinde tam kontrol

✗ Yanlış. Doğru cevap B. jOOQ bir ORM değil, SQL’i Java içinde tip güvenli olarak yazmaya yarayan bir kütüphanedir.

2. Aşağıdaki senaryolardan hangisinde jOOQ tercih edilmelidir?

✓ Doğru! Pencere fonksiyonları, karmaşık JOIN’ler ve raporlama sorguları jOOQ’nun güçlü olduğu alanlardır.

✓ Doğru! jOOQ, veritabanı şemasından Java sınıfları ürettiği için, tablo yapısı değiştiğinde kodunuz derleme aşamasında hata verir.
✗ Yanlış. Doğru cevap A. jOOQ şemadan Java sınıfları üretir; tablo yapısı değişirse kod derleme aşamasında hata verir.

10 Hibrit Yaklaşım: Hibernate + jOOQ

Aynı projede her iki aracı birlikte kullanmak mümkündür ve önerilir:

  • Yazma (Write) işlemleri: Hibernate ile (Entity yönetimi, ilişkiler, validasyon)
  • Karmaşık okuma (Read) işlemleri: jOOQ ile (Raporlar, dashboard, analitik)

Hibrit Yaklaşım Prensibi

Hibrit yaklaşımda yazma işlemleri için Hibernate’in entity yönetimi, ilişki yönetimi ve validasyon özellikleri kullanılırken; karmaşık okuma işlemleri için jOOQ’nun tip güvenli SQL yetenekleri kullanılır. Bu, her iki aracın güçlü yönlerinden faydalanmayı sağlar.

10.1 Aynı Projede Kullanım

@Service
public class ProductDashboardService {

    @Autowired
    private ProductRepository productRepository;  // Spring Data JPA (Hibernate)

    @Autowired
    private DSLContext dsl;  // jOOQ

    // Yazma islemi: Hibernate
    @Transactional
    public void createProduct(ProductCreateDTO dto) {
        Product product = new Product();
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
        productRepository.save(product);
    }

    // Karmasik okuma: jOOQ
    @Transactional(readOnly = true)
    public DashboardData getDashboard() {
        // Bu tur bir sorguyu Hibernate ile yazmak cok zahmetli olurdu
        return dsl
            .select(
                count().as("total_products"),
                sum(PRODUCTS.PRICE).as("total_value"),
                avg(PRODUCTS.PRICE).as("avg_price"),
                countDistinct(PRODUCTS.CATEGORY).as("category_count")
            )
            .from(PRODUCTS)
            .where(PRODUCTS.STATUS.eq("ACTIVE"))
            .fetchOneInto(DashboardData.class);
    }
}

10.2 jOOQ ile SQL Üretip Hibernate ile Çalıştırmak

SQL Üretim + Hibernate Çalıştırma

jOOQ’yu sadece tip güvenli SQL string’i üretmek için kullanabilir, ardından bu SQL’i Hibernate’in EntityManager’ı ile çalıştırabilirsiniz. Bu sayede sonuç, Hibernate tarafından yönetilen Managed Entity’ler olarak döner.

@Service
public class HybridQueryService {

    @PersistenceContext
    private EntityManager entityManager;

    @Autowired
    private DSLContext dsl;

    public List<Product> findComplexProducts() {
        // 1. jOOQ ile tip guvenli SQL stringi URET (calistirma)
        SelectConditionStep<?> jooqQuery = dsl
            .selectFrom(PRODUCTS)
            .where(PRODUCTS.STOCK.lt(10))
            .and(PRODUCTS.CATEGORY.eq("ELECTRONICS"));

        String sql = jooqQuery.getSQL();
        // Cikti: "SELECT * FROM products WHERE stock < ? AND category = ?"

        List<Object> bindValues = jooqQuery.getBindValues();
        // Cikti: [10, "ELECTRONICS"]

        // 2. Hibernate'in Native Query'sine ver ve calistir
        Query query = entityManager.createNativeQuery(sql, Product.class);
        for (int i = 0; i < bindValues.size(); i++) {
            query.setParameter(i + 1, bindValues.get(i));
        }

        return query.getResultList();
        // Sonuc: Hibernate tarafindan yonetilen MANAGED Entity'ler doner
    }
}
İpucu: jOOQ ile SQL üretip Hibernate ile çalıştırdığınızda, dönen nesneler Managed Entity olur. Yani üzerlerinde yapılan değişiklikler Dirty Checking ile otomatik olarak veritabanına yansır. Bu, hibrit yaklaşımın en güçlü yönlerinden biridir.

Hibrit Yaklaşımda Görev Dağılımı

Hibernate (Yazma)
  • Yeni kayıt oluşturma (CREATE)
  • Tekil kayıt güncelleme (UPDATE)
  • Silme işlemleri (DELETE)
  • Entity ilişki yönetimi
  • Validasyon ve cascade işlemleri
jOOQ (Okuma)
  • Karmaşık raporlama sorguları
  • Dashboard verileri
  • Analitik sorgular
  • Çoklu JOIN ve gruplama
  • Pencere fonksiyonları
  • Tip güvenli SQL üretimi

Bilgi Kontrolü

1. Hibrit yaklaşımda yazma işlemleri için hangi araç tercih edilir?

✓ Doğru! Yazma işlemlerinde Hibernate’in entity yönetimi, ilişki yönetimi ve validasyon özellikleri kullanılır.
✗ Yanlış. Doğru cevap B. Hibrit yaklaşımda yazma işlemleri Hibernate/Spring Data JPA ile, karmaşık okuma işlemleri jOOQ ile yapılır.

2. jOOQ ile SQL üretip Hibernate ile çalıştırdığınızda dönen nesneler hangi durumdadır?

✓ Doğru! Hibernate’in EntityManager’ı ile çalıştırılan native query’den dönen entity’ler Managed durumdadır ve Dirty Checking aktiftir.
✗ Yanlış. Doğru cevap A. Hibernate EntityManager üzerinden çalıştırılan native query’lerden dönen entity’ler Managed durumdadır.

3. Hibrit yaklaşımın en büyük avantajı nedir?

✓ Doğru! Hibrit yaklaşım, Hibernate’in entity yönetimindeki gücünü ve jOOQ’nun karmaşık sorgu yeteneklerini bir araya getirir.
✗ Yanlış. Doğru cevap C. Hibrit yaklaşım, Hibernate’in yazma tarafındaki ve jOOQ’nun okuma tarafındaki güçlü yönlerini birleştirir.
