18 Caching Tuzaklari (Hibernate 2nd Level Cache ve Spring Cache)

Hibernate Second-Level Cache Nedir?

Hibernate'in varsayilan First-Level Cache'i (Persistence Context) sadece tek bir transaction suresi boyunca yasarken, Second-Level Cache (L2) tum session'lar arasinda paylasilan, uygulama genelinde gecerli bir cache katmanidir. Amaci: Ayni Entity tekrar tekrar veritabanindan cekilmesin, cache'ten donsun.

18.1 Hibernate 2nd Level Cache Nasil Aktive Edilir?

Maven Bagimliligi (EhCache Ornegi)

<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-jcache</artifactId>
</dependency>
<dependency>
    <groupId>org.ehcache</groupId>
    <artifactId>ehcache</artifactId>
    <classifier>jakarta</classifier>
</dependency>

application.yml Konfigurasyonu

spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region.factory_class: jcache
        javax:
          cache:
            provider: org.ehcache.jsr107.EhcacheCachingProvider
      jakarta:
        persistence:
          sharedCache:
            mode: ENABLE_SELECTIVE  # Sadece @Cacheable olan Entity'ler cache'lenir

Entity Uzerinde Cache Aktive Etme

import jakarta.persistence.Cacheable;
import org.hibernate.annotations.Cache;
import org.hibernate.annotations.CacheConcurrencyStrategy;

@Entity
@Cacheable                                          // JPA standart anotasyonu
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE) // Hibernate ozel ayari
public class Country {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String code;

    // Getter/Setter...
}

18.2 Cache'in Gizli Overhead'i

Uyari: Cache "bedava hiz" degildir. Her islemde ek maliyet getirir:

  • Her READ isleminde: Once cache kontrol edilir, cache'te yoksa DB'ye gidilir, sonuc cache'e yazilir. Bu ekstra adimlar okuma islemlerini yavaslatabilir.
  • Her WRITE isleminde: DB guncellenir VE cache de guncellenir/invalidate edilir. Yazma islemleri de yavaslar.
// Cache aktif oldugunda her findById() su adimlari izler:
// 1. L2 Cache'te var mi? -> Kontrol et (ekstra islem)
// 2. Varsa: Cache'ten don (hizli!)
// 3. Yoksa: DB'den cek + Cache'e yaz (ekstra islem)
//
// Yazma isleminde:
// 1. DB'ye UPDATE yaz
// 2. L2 Cache'teki ilgili entry'yi guncelle veya invalidate et (ekstra islem)
// 3. Cluster'daki diger node'lara cache invalidation mesaji gonder (ekstra islem)

Cache Bypass ile Read Optimizasyonu

Cache'in READ overhead'ini ortadan kaldirmak icin cache'i bypass edebilirsiniz. Ancak bu karmasikligi artirir ve kodun bakimini zorlastirir.

// Cache bypass ornegi: Dogrudan DB'den oku, cache'i atla
@Transactional(readOnly = true)
public Product getProductDirectFromDb(Long id) {
    Session session = entityManager.unwrap(Session.class);
    // Cache'i bypass et, dogrudan DB'den oku
    session.setCacheMode(CacheMode.IGNORE);
    return session.get(Product.class, id);
}

18.3 Spring Boot Method-Level Caching

Spring Cache vs Hibernate L2 Cache

Hibernate L2 Cache Entity bazinda calisirken, Spring Cache metot bazinda calisir. Herhangi bir metodun donus degerini cache'leyebilirsiniz. Ikisi birbirinden bagimsizdir.

// 1. Ana sinifta cache'i aktive edin
@SpringBootApplication
@EnableCaching  // Spring Cache'i aktive eder
public class Application { ... }

// 2. Servis metodlarinda kullanin
@Service
public class ProductService {

    @Autowired private ProductRepository repository;

    // Sonuc cache'lenir. Ayni id ile tekrar cagirildiginda DB'ye gidilmez.
    @Cacheable(value = "products", key = "#id")
    @Transactional(readOnly = true)
    public ProductResponseDTO getById(Long id) {
        return repository.findById(id)
            .map(this::toDto)
            .orElseThrow();
    }

    // Entity guncellendiginde cache'teki eski veriyi temizle
    @CacheEvict(value = "products", key = "#id")
    @Transactional
    public void updateProduct(Long id, ProductPutDTO dto) {
        Product product = repository.findById(id).orElseThrow();
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
    }

    // Tum cache'i temizle (dikkatli kullanin!)
    @CacheEvict(value = "products", allEntries = true)
    public void clearAllProductCache() {
        // Cache tamamen temizlenir
    }
}

18.4 Cache Ne Zaman Kullanilmali?

DOGRU -- Statik/Nadir Degisen Veri icin Cache

// Ulke listesi, para birimi, kategori gibi nadir degisen veriler
// Cache HARIKA calisir!
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)  // Hic degismez!
public class Country {

    private String name;
    private String code;

// Konfigurasyon tablolari, enum tablolari icin idealdir.
// Bu veriler uygulama boyunca hemen hemen hic degismez.

// Siparis, stok, fiyat gibi surekli degisen veriler
// Cache TEHLIKELI! Stale data riski yuksek!
@Entity
@Cacheable  // KOTU FIKIR!
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Order {
    @Id private Long id;
    private OrderStatus status;       // Surekli degisiyor
    private BigDecimal totalAmount;   // Surekli degisiyor
    private LocalDateTime updatedAt;  // Surekli degisiyor

// Cache surekli invalidate edilecek -> overhead > kazanc

18.6 Best Practice: Cache En Son Care Olsun

Ipucu: Cache eklemeden once su adimlari izleyin:

  1. Sorguyu optimize edin: INDEX ekleyin, N+1 problemini cozun, DTO Projection kullanin
  2. Veritabanini optimize edin: Connection pool ayarlari, query plan analizi
  3. Hala yavas mi? ANCAK o zaman cache dusunun

Cache karmasiklik ekler (invalidation, stale data, cluster sync, memory yonetimi). Sorguyu optimize etmek her zaman daha basit ve surdurulebilir bir cozumdur.

Bilgi Kontrolu

1. Hibernate Second-Level Cache hangi tur veri icin en uygun secimdir?

✓ Dogru! L2 Cache statik veya nadir degisen referans verileri icin idealdir. Surekli degisen veriler icin cache overhead'i kazanctan fazla olur.
✗ Yanlis. Dogru cevap C: Cache, nadir degisen referans verileri icin idealdir. Sik degisen verilerde stale data riski ve invalidation overhead'i yuksektir.

2. Cache aktif oldugunda READ islemlerinde ne olur?

✓ Dogru! Her read isleminde once cache kontrol edilir. Cache hit varsa hizlidir, ancak cache miss durumunda hem cache kontrolu hem DB erisimi yapilir ve ekstra overhead olusur.
✗ Yanlis. Dogru cevap B: Cache her read isleminde kontrol edilir. Cache miss durumunda hem cache kontrol maliyeti hem DB erisimi birlikte olusur, bu da overhead ekler.

3. Performans problemi yasandiginda ilk yapilmasi gereken nedir?

✓ Dogru! Cache en son care olmalidir. Once sorguyu optimize edin, INDEX ekleyin, N+1 problemini cozun. Cache karmasiklik ekler, sorgu optimizasyonu daha basit ve surdurulebilirdir.
✗ Yanlis. Dogru cevap A: Once sorguyu optimize etmek gerekir. Cache karmasiklik ekler ve bakim maliyeti yuksektir. Sorgu optimizasyonu her zaman oncelikli olmalidir.

19 SQL Ogrenmek Neden Kritik?

Thorben'in Ana Mesaji: "Framework SQL'i benden gizliyor, SQL ogrenmeme gerek yok" dusuncesi production ortaminda patlamalarin en buyuk sebeplerinden biridir. Hibernate/Spring Data iyi SQL uretir, AMA gelistirici yanlis input verirse (yanlis entity mapping, yanlis iliski tanimlari) uretilen SQL de kotu olur.

19.1 Framework Suclama Sendromu

"Framework Kotu SQL Uretiyor" Efsanesi

Gelistiriciler sik sik "Hibernate yavas SQL uretiyor" derler. Gercekte sorun Hibernate degil, yanlis konfigurasyondur. Framework'u suclamak yerine, framework'un nasil calistigini ogrenmeniz ve dogru input vermeniz gerekir.

YANLIS -- Kotu Entity Mapping (Kotu SQL Uretir)

@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // HATA 1: FetchType.EAGER -- Her Order sorgulandiginda
    // TUM item'lar otomatik cekilir (gereksiz JOIN veya N+1)
    @OneToMany(mappedBy = "order", fetch = FetchType.EAGER)
    private List<OrderItem> items;

    // HATA 2: FetchType.EAGER -- Her Order sorgulandiginda
    // musteri bilgisi de cekilir
    @ManyToOne(fetch = FetchType.EAGER)
    private Customer customer;

    // HATA 3: FetchType.EAGER -- Odeme bilgisi de cekilir
    @OneToOne(fetch = FetchType.EAGER)
    private Payment payment;
}

// SONUC: Basit bir "SELECT * FROM orders" sorgusu bile
// arka planda 4 tablo JOIN'liyor veya N+1 sorgu atiyor!
// "Hibernate yavas" degil, SEN yanlis mapping yaptin!

DOGRU -- Dogru Entity Mapping (Verimli SQL Uretir)

@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // DOGRU: LAZY -- Sadece eristiginde yuklenir
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;

    // DOGRU: LAZY -- Sadece gerektiginde yuklenir
    @ManyToOne(fetch = FetchType.LAZY)
    private Customer customer;

    // DOGRU: LAZY -- Sadece gerektiginde yuklenir
    @OneToOne(fetch = FetchType.LAZY)
    private Payment payment;
}

// Ihtiyac oldugunda JOIN FETCH ile cek:
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id")
Optional<Order> findWithItems(@Param("id") Long id);

// Sadece ozet bilgi lazimsa DTO Projection kullan:
@Query("SELECT new com.example.dto.OrderSummary(o.id, o.status, o.createdAt) " +
       "FROM Order o WHERE o.status = :status")
List<OrderSummary> findSummaries(@Param("status") OrderStatus status);

19.2 Temel SQL Bilgisi (1-3 Gunde Ogrenilebilir)

Her Java Gelistiricisinin Bilmesi Gereken SQL

Asagidaki temel SQL bilgisi 1-3 gun icinde ogrenilebilir ve Hibernate'in urettigi SQL'i anlamaniz icin yeterlidir:

-- 1. INSERT: Yeni kayit ekleme
INSERT INTO products (name, price, category)
VALUES ('Laptop', 15000.00, 'Elektronik');

-- 2. SELECT: Veri sorgulama
SELECT p.name, p.price
FROM products p
WHERE p.category = 'Elektronik'
ORDER BY p.price DESC;

-- 3. JOIN: Tablolari birlestirme
-- Hibernate'in urettigi SQL'leri anlamak icin JOIN bilmek SART!
SELECT o.id, o.status, c.name AS customer_name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'ACTIVE';

-- 4. INDEX: Sorgu performansini artirma
-- Hibernate INDEX olusturmaz! Bunu SIZ yapmaniz gerekir.
CREATE INDEX idx_products_category ON products (category);
CREATE INDEX idx_orders_status ON orders (status);
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

19.3 Ileri SQL (Zamanla Ogrenilecek)

-- Window Functions: Siralama, gruplama icinde hesaplama
SELECT name, price, category,
       RANK() OVER (PARTITION BY category ORDER BY price DESC) AS rank_in_category
FROM products;

-- CTE (Common Table Expression): Karmasik sorulari parcalama
WITH expensive_products AS (
    SELECT * FROM products WHERE price > 10000
)
SELECT category, COUNT(*) AS count, AVG(price) AS avg_price
FROM expensive_products
GROUP BY category;

-- Karmasik Subquery
SELECT *
FROM orders o
WHERE o.total_amount > (
    SELECT AVG(total_amount) FROM orders WHERE status = 'COMPLETED'
);

19.4 Object-Relational Impedance Mismatch

Nesne-Tablo Uyumsuzlugu

Java'daki nesne modeli ile SQL'deki iliskisel model arasinda temel farkliliklar vardir. Bu uyumsuzluk ORM'lerin varlik sebebidir, ama ayni zamanda en buyuk tuzak kaynagidir.

Ozellik Java (Nesne Modeli) SQL (Iliskisel Model)
Kalitim (Inheritance) Dogal: class Dog extends Animal Yok! Farkli stratejiler gerekir (SINGLE_TABLE, JOINED, TABLE_PER_CLASS)
Kimlik == ve equals() farki PRIMARY KEY
Iliskiler Referans: order.getCustomer() Foreign Key + JOIN
Veri Erisimi Navigasyon: a.getB().getC() JOIN ile duz tablo sonucu 
// Java'da kalitim dogal:
public abstract class Payment { ... }
public class CreditCardPayment extends Payment { ... }
public class BankTransferPayment extends Payment { ... }

// SQL'de kalitim YOKTUR! Hibernate 3 strateji sunar:
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE) // En hizli, ama null alanlar olusur
// veya
@Inheritance(strategy = InheritanceType.JOINED)       // Normalizasyon iyi, ama JOIN maliyeti
// veya
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS) // Her sinif ayri tablo, UNION gerekir

Ipucu: SQL ogrenmek icin harcayacaginiz 1-3 gun, production'da haftalar surecek debug suresinden sizi kurtarir. Hibernate'in urettigi SQL'i okuyamazsaniz, performans problemlerini teshis edemezsiniz.

Bilgi Kontrolu

1. "Hibernate kotu SQL uretiyor" sikayetinin gercek sebebi genellikle nedir?

✓ Dogru! Hibernate iyi SQL uretir, ancak yanlis mapping (FetchType.EAGER, eksik index, yanlis iliski tanimlari) kotu SQL'e yol acar. Sorun framework degil, framework'a verilen inputtur.
✗ Yanlis. Dogru cevap B: Hibernate'in urettigi SQL, gelistiricinin verdigi mapping'e baglidir. Yanlis mapping = kotu SQL. Framework'u suclamak yerine dogru mapping ogrenilmelidir.

2. Object-Relational Impedance Mismatch'in en belirgin ornegi nedir?

✓ Dogru! Java'da class Dog extends Animal yazmak dogaldir, ancak SQL'de kalitim yoktur. Hibernate bunu SINGLE_TABLE, JOINED veya TABLE_PER_CLASS stratejileriyle simule eder ve her birinin trade-off'lari vardir.
✗ Yanlis. Dogru cevap A: Inheritance Java'da dogal bir ozellikken SQL'de boyle bir kavram yoktur. Bu uyumsuzluk ORM'lerin en zorlayici noktalarindan biridir.

3. Asagidakilerden hangisi temel SQL bilgisi kapsaminda 1-3 gunde ogrenilebilecek konular arasindadir?

✓ Dogru! INSERT, SELECT, JOIN ve INDEX temel SQL bilgisidir ve kisa surede ogrenilebilir. Window functions ve CTE'ler ise ileri seviye konularidir.
✗ Yanlis. Dogru cevap C: INSERT, SELECT, JOIN ve INDEX kavramlari temel SQL bilgisidir ve 1-3 gunde ogrenilebilir. Window functions ve CTE'ler ileri seviye konulardir.

20 Gelistirme Sirasinda SQL Izleme

Thorben'in En Onemli Tavsiyesi: "SQL statement'lari development sirasinda MUTLAKA izleyin. Production'da performans problemi yasiyorsaniz ve development'ta hic SQL izlemediyseniz, sikayetci olma hakkiniz yoktur."

20.1 Hibernate show-sql Ayari (Temel Yontem)

En Basit Yol: show-sql

Hibernate'in urettigi tum SQL'leri konsola yazdirmanin en kolay yolu show-sql: true ayaridir. Ancak bu yontem parametreleri gostermez ve ciktiyi formatlamaz.

# application.yml -- Temel SQL gosterme
spring:
  jpa:
    show-sql: true          # SQL'leri konsola yazdir
    properties:
      hibernate:
        format_sql: true    # SQL'leri okunabilir formatta yazdir

Ipucu: show-sql SQL'leri System.out'a yazdirir (loglama framework'unu bypass eder). Daha iyi kontrol icin loglama seviyelerini kullanin (Bolum 20.2).

20.2 Log Seviyelerini Ayarlama (Onerilen Yontem)

# application.yml -- Log seviyesi ile SQL izleme (ONERILEN)
spring:
  jpa:
    show-sql: false   # System.out yerine loglama kullan

logging:
  level:
    # Hibernate'in urettigi SQL'leri gor
    org.hibernate.SQL: DEBUG

    # SQL parametrelerini gor (? yerine gercek degerleri goster)
    org.hibernate.orm.jdbc.bind: TRACE

    # Transaction islemlerini izle
    org.hibernate.engine.transaction: DEBUG

20.4 p6spy ile Detayli SQL Loglama

p6spy Nedir?

p6spy, JDBC driver'inizi sarmalayarak (wrap ederek) her SQL sorgusunu, parametreleriyle birlikte loglar. Hibernate'in gostermedigi detaylari (gercek parametre degerleri, calisma suresi) gosterir.

Maven Bagimliligi

<dependency>
    <groupId>com.github.gavlyukovskiy</groupId>
    <artifactId>p6spy-spring-boot-starter</artifactId>
    <version>1.9.1</version>
    <scope>runtime</scope>  <!-- Sadece runtime'da gerekli -->
</dependency>

application.yml Konfigurasyonu

# p6spy konfigurasyonu
decorator:
  datasource:
    p6spy:
      enable-logging: true
      multiline: true        # SQL'i formatlayarak goster
      logging: slf4j         # Loglama framework'u ile entegre
      log-format: "\n\tExecution Time: %(executionTime)ms\n\tSQL: %(sqlSingleLine)"

Cikti Ornegi

    Execution Time: 3ms
    SQL: select p.id, p.name, p.price, p.category from products p where p.category = 'Elektronik'

    Execution Time: 1ms
    SQL: select o.id, o.status from orders o where o.product_id = 42

    Execution Time: 15ms  <-- YAVAS! Neden?
    SQL: select * from order_items oi where oi.order_id IN (1,2,3,...,100)

20.6 Slow Query Detection (Yavas Sorgu Tespiti)

Uyari: Development ortaminda yavas sorgu tespitini aktive edin. Production'a gitmeden once sorunlu sorgulari yakalayabilirsiniz.

# Hibernate yerlesik slow query tespiti
spring:
  jpa:
    properties:
      hibernate:
        session:
          events:
            log:
              LOG_QUERIES_SLOWER_THAN_MS: 25  # 25ms'den yavas sorgulari logla
// Programatik slow query loglama (Custom Interceptor)
@Component
public class QueryCountInterceptor implements HandlerInterceptor {

    // Her HTTP request icin kac SQL sorgusu calistigini say
    // Eger bir endpoint 10'dan fazla sorgu atiyorsa, N+1 problemi olabilir!

    @Override
    public void afterCompletion(HttpServletRequest request,
                                 HttpServletResponse response,
                                 Object handler, Exception ex) {
        Statistics stats = entityManagerFactory.unwrap(SessionFactory.class)
            .getStatistics();
        long queryCount = stats.getQueryExecutionCount();

        if (queryCount > 10) {
            log.warn("UYARI: {} endpoint'i {} sorgu calistirdi!",
                request.getRequestURI(), queryCount);
        }
    }
}

Bilgi Kontrolu

1. Hibernate SQL loglamasinda parametrelerin gercek degerlerini gormek icin hangi ayar kullanilir?

✓ Dogru! org.hibernate.orm.jdbc.bind: TRACE ayari SQL parametrelerinin gercek degerlerini (? yerine) gosterir. show-sql sadece SQL'i gosterir, parametreleri gostermez.
✗ Yanlis. Dogru cevap B: org.hibernate.orm.jdbc.bind: TRACE ayari parametrelerin gercek degerlerini gosterir. show-sql parametreleri gostermez, format_sql ise sadece formatlama yapar.

2. Hibernate Statistics ile asagidakilerden hangisi tespit edilebilir?

✓ Dogru! Hibernate Statistics toplam JDBC statement sayisini, cache hit/miss oranlarini ve harcanan zamani gosterir. Bir islem icin beklenenden fazla sorgu calisiyorsa N+1 problemi vardir.
