18 Caching Tuzakları (Hibernate 2nd Level Cache ve Spring Cache)

Hibernate Second-Level Cache Nedir?

Hibernate'in varsayılan First-Level Cache'i (Persistence Context) sadece tek bir transaction süresi boyunca yaşarken, Second-Level Cache (L2) tüm session'lar arasında paylaşılan, uygulama genelinde geçerli bir cache katmanıdır. Amacı: Aynı Entity tekrar tekrar veritabanından çekilmesin, cache'ten dönsün.

18.1 Hibernate 2nd Level Cache Nasıl Aktive Edilir?

Maven Bağımlılığı (EhCache Örneği)

<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-jcache</artifactId>
</dependency>
<dependency>
    <groupId>org.ehcache</groupId>
    <artifactId>ehcache</artifactId>
    <classifier>jakarta</classifier>
</dependency>

application.yml Konfigürasyonu

spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region.factory_class: jcache
        javax:
          cache:
            provider: org.ehcache.jsr107.EhcacheCachingProvider
      jakarta:
        persistence:
          sharedCache:
            mode: ENABLE_SELECTIVE  # Sadece @Cacheable olan Entity'ler cache'lenir

Entity Üzerinde Cache Aktive Etme

import jakarta.persistence.Cacheable;
import org.hibernate.annotations.Cache;
import org.hibernate.annotations.CacheConcurrencyStrategy;

@Entity
@Cacheable                                          // JPA standart anotasyonu
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE) // Hibernate özel ayarı
public class Country {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String code;

    // Getter/Setter...
}

18.2 Cache'in Gizli Overhead'i

Uyarı: Cache "bedava hız" değildir. Her işlemde ek maliyet getirir:

  • Her READ işleminde: Önce cache kontrol edilir, cache'te yoksa DB'ye gidilir, sonuç cache'e yazılır. Bu ekstra adımlar okuma işlemlerini yavaşlatabilir.
  • Her WRITE işleminde: DB güncellenir VE cache de güncellenir/invalidate edilir. Yazma işlemleri de yavaşlar.
// Cache aktif olduğunda her findById() şu adımları izler:
// 1. L2 Cache'te var mı? -> Kontrol et (ekstra işlem)
// 2. Varsa: Cache'ten dön (hızlı!)
// 3. Yoksa: DB'den çek + Cache'e yaz (ekstra işlem)
//
// Yazma işleminde:
// 1. DB'ye UPDATE yaz
// 2. L2 Cache'teki ilgili entry'yi güncelle veya invalidate et (ekstra işlem)
// 3. Cluster'daki diğer node'lara cache invalidation mesajı gönder (ekstra işlem)

Cache Bypass ile Read Optimizasyonu

Cache'in READ overhead'ini ortadan kaldırmak için cache'i bypass edebilirsiniz. Ancak bu karmaşıklığı artırır ve kodun bakımını zorlaştırır.

// Cache bypass örneği: Doğrudan DB'den oku, cache'i atla
@Transactional(readOnly = true)
public Product getProductDirectFromDb(Long id) {
    Session session = entityManager.unwrap(Session.class);
    // Cache'i bypass et, doğrudan DB'den oku
    session.setCacheMode(CacheMode.IGNORE);
    return session.get(Product.class, id);
}

18.3 Spring Boot Method-Level Caching

Spring Cache vs Hibernate L2 Cache

Hibernate L2 Cache Entity bazında çalışırken, Spring Cache metot bazında çalışır. Herhangi bir metodun dönüş değerini cache'leyebilirsiniz. İkisi birbirinden bağımsızdır.

// 1. Ana sınıfta cache'i aktive edin
@SpringBootApplication
@EnableCaching  // Spring Cache'i aktive eder
public class Application { ... }

// 2. Servis metotlarında kullanın
@Service
public class ProductService {

    @Autowired private ProductRepository repository;

    // Sonuç cache'lenir. Aynı id ile tekrar çağırıldığında DB'ye gidilmez.
    @Cacheable(value = "products", key = "#id")
    @Transactional(readOnly = true)
    public ProductResponseDTO getById(Long id) {
        return repository.findById(id)
            .map(this::toDto)
            .orElseThrow();
    }

    // Entity güncellendiğinde cache'teki eski veriyi temizle
    @CacheEvict(value = "products", key = "#id")
    @Transactional
    public void updateProduct(Long id, ProductPutDTO dto) {
        Product product = repository.findById(id).orElseThrow();
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
    }

    // Tüm cache'i temizle (dikkatli kullanın!)
    @CacheEvict(value = "products", allEntries = true)
    public void clearAllProductCache() {
        // Cache tamamen temizlenir
    }
}

18.4 Cache Ne Zaman Kullanılmalı?

DOĞRU -- Statik/Nadir Değişen Veri için Cache

// Ülke listesi, para birimi, kategori gibi nadir değişen veriler
// Cache HARIKA çalışır!
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)  // Hiç değişmez!
public class Country {
    @Id private Long id;
    private String name;
    private String code;
}

// Konfigürasyon tabloları, enum tabloları için idealdir.
// Bu veriler uygulama boyunca hemen hemen hiç değişmez.

YANLIŞ -- Sık Değişen Veri için Cache

// Sipariş, stok, fiyat gibi sürekli değişen veriler
// Cache TEHLİKELİ! Stale data riski yüksek!
@Entity
@Cacheable  // KÖTÜ FİKİR!
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Order {
    @Id private Long id;
    private OrderStatus status;       // Sürekli değişiyor
    private BigDecimal totalAmount;   // Sürekli değişiyor
    private LocalDateTime updatedAt;  // Sürekli değişiyor
}
// Cache sürekli invalidate edilecek -> overhead > kazanç
// Ayrıca cluster ortamında stale data riski!

18.5 Startup Time Problemi

Özellikle Kubernetes gibi ortamlarda pod'lar hızla ayağa kalkmalı -- uzun startup süreleri health check hatası ve restart döngüsü oluşturabilir.

18.6 Best Practice: Cache En Son Çare Olsun

İpucu: Cache eklemeden önce şu adımları izleyin:

  1. Sorguyu optimize edin: INDEX ekleyin, N+1 problemini çözün, DTO Projection kullanın
  2. Veritabanını optimize edin: Connection pool ayarları, query plan analizi
  3. Hala yavaş mı? ANCAK o zaman cache düşünün

Cache karmaşıklık ekler (invalidation, stale data, cluster sync, memory yönetimi). Sorguyu optimize etmek her zaman daha basit ve sürdürülebilir bir çözümdür.

Bilgi Kontrolü

1. Hibernate Second-Level Cache hangi tür veri için en uygun seçimdir?

✓ Doğru! L2 Cache statik veya nadir değişen referans verileri için idealdir. Sürekli değişen veriler için cache overhead'i kazançtan fazla olur.
✗ Yanlış. Doğru cevap C: Cache, nadir değişen referans verileri için idealdir. Sık değişen verilerde stale data riski ve invalidation overhead'i yüksektir.

2. Cache aktif olduğunda READ işlemlerinde ne olur?

✓ Doğru! Her read işleminde önce cache kontrol edilir. Cache hit varsa hızlıdır, ancak cache miss durumunda hem cache kontrolü hem DB erişimi yapılır ve ekstra overhead oluşur.
✗ Yanlış. Doğru cevap B: Cache her read işleminde kontrol edilir. Cache miss durumunda hem cache kontrol maliyeti hem DB erişimi birlikte oluşur, bu da overhead ekler.

3. Performans problemi yaşandığında ilk yapılması gereken nedir?

✓ Doğru! Cache en son çare olmalıdır. Önce sorguyu optimize edin, INDEX ekleyin, N+1 problemini çözün. Cache karmaşıklık ekler, sorgu optimizasyonu daha basit ve sürdürülebilirdir.
✗ Yanlış. Doğru cevap A: Önce sorguyu optimize etmek gerekir. Cache karmaşıklık ekler ve bakım maliyeti yüksektir. Sorgu optimizasyonu her zaman öncelikli olmalıdır.

19 SQL Öğrenmek Neden Kritik?

Thorben'in Ana Mesajı: "Framework SQL'i benden gizliyor, SQL öğrenmeme gerek yok" düşüncesi production ortamında patlamaların en büyük sebeplerinden biridir. Hibernate/Spring Data iyi SQL üretir, AMA geliştirici yanlış input verirse (yanlış entity mapping, yanlış ilişki tanımları) üretilen SQL de kötü olur.

19.1 Framework Suçlama Sendromu

"Framework Kötü SQL Üretiyor" Efsanesi

Geliştiriciler sık sık "Hibernate yavaş SQL üretiyor" derler. Gerçekte sorun Hibernate değil, yanlış konfigürasyondur. Framework'u suçlamak yerine, framework'un nasıl çalıştığını öğrenmeniz ve doğru input vermeniz gerekir.

YANLIŞ -- Kötü Entity Mapping (Kötü SQL Üretir)

@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // HATA 1: FetchType.EAGER -- Her Order sorgulandığında
    // TÜM item'lar otomatik çekilir (gereksiz JOIN veya N+1)
    @OneToMany(mappedBy = "order", fetch = FetchType.EAGER)
    private List<OrderItem> items;

    // HATA 2: FetchType.EAGER -- Her Order sorgulandığında
    // müşteri bilgisi de çekilir
    @ManyToOne(fetch = FetchType.EAGER)
    private Customer customer;

    // HATA 3: FetchType.EAGER -- Ödeme bilgisi de çekilir
    @OneToOne(fetch = FetchType.EAGER)
    private Payment payment;
}

// SONUÇ: Basit bir "SELECT * FROM orders" sorgusu bile
// arka planda 4 tablo JOIN'liyor veya N+1 sorgu atıyor!
// "Hibernate yavaş" değil, SEN yanlış mapping yaptın!

DOĞRU -- Doğru Entity Mapping (Verimli SQL Üretir)

@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // DOĞRU: LAZY -- Sadece eriştiğinde yüklenir
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;

    // DOĞRU: LAZY -- Sadece gerektiğinde yüklenir
    @ManyToOne(fetch = FetchType.LAZY)
    private Customer customer;

    // DOĞRU: LAZY -- Sadece gerektiğinde yüklenir
    @OneToOne(fetch = FetchType.LAZY)
    private Payment payment;
}

// İhtiyaç olduğunda JOIN FETCH ile çek:
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id")
Optional<Order> findWithItems(@Param("id") Long id);

// Sadece özet bilgi lazımsa DTO Projection kullan:
@Query("SELECT new com.example.dto.OrderSummary(o.id, o.status, o.createdAt) " +
       "FROM Order o WHERE o.status = :status")
List<OrderSummary> findSummaries(@Param("status") OrderStatus status);

19.2 Temel SQL Bilgisi (1-3 Günde Öğrenilebilir)

Her Java Geliştiricisinin Bilmesi Gereken SQL

Aşağıdaki temel SQL bilgisi 1-3 gün içinde öğrenilebilir ve Hibernate'in ürettiği SQL'i anlamanız için yeterlidir:

-- 1. INSERT: Yeni kayıt ekleme
INSERT INTO products (name, price, category)
VALUES ('Laptop', 15000.00, 'Elektronik');

-- 2. SELECT: Veri sorgulama
SELECT p.name, p.price
FROM products p
WHERE p.category = 'Elektronik'
ORDER BY p.price DESC;

-- 3. JOIN: Tabloları birleştirme
-- Hibernate'in ürettiği SQL'leri anlamak için JOIN bilmek ŞART!
SELECT o.id, o.status, c.name AS customer_name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'ACTIVE';

-- 4. INDEX: Sorgu performansını artırma
-- Hibernate INDEX oluşturmaz! Bunu SİZ yapmanız gerekir.
CREATE INDEX idx_products_category ON products (category);
CREATE INDEX idx_orders_status ON orders (status);
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

19.3 İleri SQL (Zamanla Öğrenilecek)

-- Window Functions: Sıralama, gruplama içinde hesaplama
SELECT name, price, category,
       RANK() OVER (PARTITION BY category ORDER BY price DESC) AS rank_in_category
FROM products;

-- CTE (Common Table Expression): Karmaşık soruları parçalama
WITH expensive_products AS (
    SELECT * FROM products WHERE price > 10000
)
SELECT category, COUNT(*) AS count, AVG(price) AS avg_price
FROM expensive_products
GROUP BY category;

-- Karmaşık Subquery
SELECT *
FROM orders o
WHERE o.total_amount > (
    SELECT AVG(total_amount) FROM orders WHERE status = 'COMPLETED'
);

19.4 Object-Relational Impedance Mismatch

Nesne-Tablo Uyumsuzluğu

Java'daki nesne modeli ile SQL'deki ilişkisel model arasında temel farklılıklar vardır. Bu uyumsuzluk ORM'lerin varlık sebebidir, ama aynı zamanda en büyük tuzak kaynağıdır.

Özellik Java (Nesne Modeli) SQL (İlişkisel Model)
Kalıtım (Inheritance) Doğal: class Dog extends Animal Yok! Farklı stratejiler gerekir (SINGLE_TABLE, JOINED, TABLE_PER_CLASS)
Kimlik == ve equals() farkı PRIMARY KEY
İlişkiler Referans: order.getCustomer() Foreign Key + JOIN
Veri Erişimi Navigasyon: a.getB().getC() JOIN ile düz tablo sonucu 
// Java'da kalıtım doğal:
public abstract class Payment { ... }
public class CreditCardPayment extends Payment { ... }
public class BankTransferPayment extends Payment { ... }

// SQL'de kalıtım YOKTUR! Hibernate 3 strateji sunar:
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE) // En hızlı, ama null alanlar oluşur
// veya
@Inheritance(strategy = InheritanceType.JOINED)       // Normalizasyon iyi, ama JOIN maliyeti
// veya
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS) // Her sınıf ayrı tablo, UNION gerekir

İpucu: SQL öğrenmek için harcayacağınız 1-3 gün, production'da haftalar sürecek debug süresinden sizi kurtarır. Hibernate'in ürettiği SQL'i okuyamazsanız, performans problemlerini teşhis edemezsiniz.

Bilgi Kontrolü

1. "Hibernate kötü SQL üretiyor" şikayetinin gerçek sebebi genellikle nedir?

✓ Doğru! Hibernate iyi SQL üretir, ancak yanlış mapping (FetchType.EAGER, eksik index, yanlış ilişki tanımları) kötü SQL'e yol açar. Sorun framework değil, framework'a verilen inputtur.
✗ Yanlış. Doğru cevap B: Hibernate'in ürettiği SQL, geliştiricinin verdiği mapping'e bağlıdır. Yanlış mapping = kötü SQL. Framework'u suçlamak yerine doğru mapping öğrenilmelidir.

2. Object-Relational Impedance Mismatch'in en belirgin örneği nedir?

✓ Doğru! Java'da class Dog extends Animal yazmak doğaldır, ancak SQL'de kalıtım yoktur. Hibernate bunu SINGLE_TABLE, JOINED veya TABLE_PER_CLASS stratejileriyle simüle eder ve her birinin trade-off'ları vardır.
✗ Yanlış. Doğru cevap A: Inheritance Java'da doğal bir özellikken SQL'de böyle bir kavram yoktur. Bu uyumsuzluk ORM'lerin en zorlayıcı noktalarından biridir.

3. Aşağıdakilerden hangisi temel SQL bilgisi kapsamında 1-3 günde öğrenilebilecek konular arasındadır?

✓ Doğru! INSERT, SELECT, JOIN ve INDEX temel SQL bilgisidir ve kısa sürede öğrenilebilir. Window functions ve CTE'ler ise ileri seviye konularıdır.
✗ Yanlış. Doğru cevap C: INSERT, SELECT, JOIN ve INDEX kavramları temel SQL bilgisidir ve 1-3 günde öğrenilebilir. Window functions ve CTE'ler ileri seviye konulardır.

20 Geliştirme Sırasında SQL İzleme

Thorben'in En Önemli Tavsiyesi: "SQL statement'ları development sırasında MUTLAKA izleyin. Production'da performans problemi yaşıyorsanız ve development'ta hiç SQL izlemediyseniz, şikayetçi olma hakkınız yoktur."

20.1 Hibernate show-sql Ayarı (Temel Yöntem)

En Basit Yol: show-sql

Hibernate'in ürettiği tüm SQL'leri konsola yazdırmanın en kolay yolu show-sql: true ayarıdır. Ancak bu yöntem parametreleri göstermez ve çıktıyı formatlamaz.

# application.yml -- Temel SQL gösterme
spring:
  jpa:
    show-sql: true          # SQL'leri konsola yazdır
    properties:
      hibernate:
        format_sql: true    # SQL'leri okunabilir formatta yazdır

İpucu: show-sql SQL'leri System.out'a yazdırır (loglama framework'unu bypass eder). Daha iyi kontrol için loglama seviyelerini kullanın (Bölüm 20.2).

20.2 Log Seviyelerini Ayarlama (Önerilen Yöntem)

# application.yml -- Log seviyesi ile SQL izleme (ÖNERİLEN)
spring:
  jpa:
    show-sql: false   # System.out yerine loglama kullan

logging:
  level:
    # Hibernate'in ürettiği SQL'leri gör
    org.hibernate.SQL: DEBUG

    # SQL parametrelerini gör (? yerine gerçek değerleri göster)
    org.hibernate.orm.jdbc.bind: TRACE

    # Transaction işlemlerini izle
    org.hibernate.engine.transaction: DEBUG

20.3 Hibernate Statistics (İstatistikler)

Hibernate Statistics Nedir?

Hibernate, bir transaction veya session içinde kaç sorgu çalıştığını, kaç cache hit olduğunu ve ne kadar zaman harcandığını ölçebilir. N+1 problemini tespit etmenin en kolay yoludur.

# application.yml -- Hibernate istatistiklerini aktive et
spring:
  jpa:
    properties:
      hibernate:
        generate_statistics: true

logging:
  level:
    org.hibernate.stat: DEBUG

Çıktı Örneği

-- Session Metrics --
    242700 nanoseconds spent acquiring 1 JDBC connections;
    0 nanoseconds spent releasing 0 JDBC connections;
    505200 nanoseconds spent preparing 6 JDBC statements;
    2025400 nanoseconds spent executing 6 JDBC statements;
    0 nanoseconds spent executing 0 JDBC batches;
    0 nanoseconds spent performing 0 L2C puts;
    0 nanoseconds spent performing 0 L2C hits;
    0 nanoseconds spent performing 0 L2C misses;

-- DİKKAT: 6 JDBC statements! Belki de 1 olmalı?
-- Bu N+1 probleminin açık bir işaretidir!

20.4 p6spy ile Detaylı SQL Loglama

p6spy Nedir?

p6spy, JDBC driver'inizi sarmalayarak (wrap ederek) her SQL sorgusunu, parametreleriyle birlikte loglar. Hibernate'in göstermediği detayları (gerçek parametre değerleri, çalışma süresi) gösterir.

Maven Bağımlılığı

<dependency>
    <groupId>com.github.gavlyukovskiy</groupId>
    <artifactId>p6spy-spring-boot-starter</artifactId>
    <version>1.9.1</version>
    <scope>runtime</scope>  <!-- Sadece runtime'da gerekli -->
</dependency>

application.yml Konfigürasyonu

# p6spy konfigürasyonu
decorator:
  datasource:
    p6spy:
      enable-logging: true
      multiline: true        # SQL'i formatlayarak göster
      logging: slf4j         # Loglama framework'u ile entegre
      log-format: "\n\tExecution Time: %(executionTime)ms\n\tSQL: %(sqlSingleLine)"

Çıktı Örneği

    Execution Time: 3ms
    SQL: select p.id, p.name, p.price, p.category from products p where p.category = 'Elektronik'

    Execution Time: 1ms
    SQL: select o.id, o.status from orders o where o.product_id = 42

    Execution Time: 15ms  <-- YAVAŞ! Neden?
    SQL: select * from order_items oi where oi.order_id IN (1,2,3,...,100)

20.6 Slow Query Detection (Yavaş Sorgu Tespiti)

Uyarı: Development ortamında yavaş sorgu tespitini aktive edin. Production'a gitmeden önce sorunlu sorguları yakalayabilirsiniz.

# Hibernate yerleşik slow query tespiti
spring:
  jpa:
    properties:
      hibernate:
        session:
          events:
            log:
              LOG_QUERIES_SLOWER_THAN_MS: 25  # 25ms'den yavaş sorguları logla
// Programatik slow query loglama (Custom Interceptor)
@Component
public class QueryCountInterceptor implements HandlerInterceptor {

    // Her HTTP request için kaç SQL sorgusu çalıştığını say
    // Eğer bir endpoint 10'dan fazla sorgu atıyorsa, N+1 problemi olabilir!

    @Override
    public void afterCompletion(HttpServletRequest request,
                                 HttpServletResponse response,
                                 Object handler, Exception ex) {
        Statistics stats = entityManagerFactory.unwrap(SessionFactory.class)
            .getStatistics();
        long queryCount = stats.getQueryExecutionCount();

        if (queryCount > 10) {
            log.warn("UYARI: {} endpoint'i {} sorgu çalıştırdı!",
                request.getRequestURI(), queryCount);
        }
    }
}
