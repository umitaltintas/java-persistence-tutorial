14 REST API'lerde PUT ve PATCH Best Practices

14.1 PUT vs PATCH Farki

Ozellik PUT (Tam Degistirme) PATCH (Kismi Guncelleme)
Anlam "Gonderdigimle tamamen degistir" "Sadece gonderdigim alanlari guncelle"
Gelen JSON TUM alanlar gonderilmeli Sadece degisecek alanlar
Gonderilmeyen alan null / varsayilan olur Mevcut degeri korur

14.2 Altin Kural: Entity'yi Controller'da KULLANMA!

Entity vs DTO Kurali

Entity'yi dogrudan JSON'a baglamak guvenlik acigi olusturur. Kullanici ID, createdAt gibi alanlari degistirebilir. Her zaman DTO kullanin.

YANLIS -- Entity'yi dogrudan JSON'a baglamak

@PutMapping("/{id}")
public void update(@PathVariable Long id, @RequestBody Product product) {
    // product DETACHED! Persistence Context'te degil!
    // Guvenlik acigi: Kullanici ID, createdAt gibi alanlari degistirebilir!
    productRepository.save(product);  // Tehlikeli!
}

DOGRU -- DTO kullan

@PutMapping("/{id}")
public ResponseEntity<Void> update(@PathVariable Long id,
                                   @Valid @RequestBody ProductPutDTO dto) {
    productService.updateProduct(id, dto);
    return ResponseEntity.noContent().build();
}

14.3 PUT Islemi (Tam Degistirme)

// DTO
public class ProductPutDTO {
    @NotBlank
    private String name;
    @NotNull
    private BigDecimal price;
    private String description;  // null gelebilir -- PUT'ta sorun degil
    @NotNull
    private String category;
    // Getter/Setter...
}

// Servis
@Service
public class ProductService {

    @Autowired
    private ProductRepository repository;

    @Transactional
    public void updateProductPut(Long id, ProductPutDTO dto) {
        // 1. Fetch -- Nesne MANAGED olur
        Product product = repository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Urun bulunamadi: " + id));

        // 2. Tum alanlari ezerek yaz (overwrite)
        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
        product.setDescription(dto.getDescription());  // null geldiyse DB'de de null olur
        product.setCategory(dto.getCategory());

        // 3. save() YOK -- Managed Entity, Dirty Checking yapacak
    }
}

Ipucu: PUT isleminde save() cagrisi yapmaniza gerek yoktur. Entity managed durumda oldugu icin Dirty Checking mekanizmasi degisiklikleri otomatik olarak veritabanina yansitir.

14.4 PATCH Islemi -- Yontem 1: Map<String, Object>

Map Yaklasimi

"Gonderilmeyen alan" ile "null gonderilen alan" ayrimini yapmanin en basit yolu Map kullanmaktir. JSON'da gonderilen anahtarlari containsKey() ile kontrol edersiniz.

// Controller
@PatchMapping("/{id}")
public ResponseEntity<Void> patch(@PathVariable Long id,
                                  @RequestBody Map<String, Object> updates) {
    productService.patchProduct(id, updates);
    return ResponseEntity.noContent().build();
}

// Servis
@Service
public class ProductService {

    @Transactional
    public void patchProduct(Long id, Map<String, Object> updates) {
        Product product = repository.findById(id).orElseThrow();

        // Sadece JSON'da GONDERILEN anahtarlari guncelle
        if (updates.containsKey("name")) {
            product.setName((String) updates.get("name"));
        }
        if (updates.containsKey("price")) {
            product.setPrice(new BigDecimal(updates.get("price").toString()));
        }
        if (updates.containsKey("description")) {
            product.setDescription((String) updates.get("description"));
            // null geldiyse: client bilerek temizlemek istiyor -- DB'de null olur
        }
        if (updates.containsKey("category")) {
            product.setCategory((String) updates.get("category"));
        }

        // save() YOK -- Managed Entity
    }
}

14.5 PATCH Islemi -- Yontem 2: DTO + Tip Guvenligi

// PATCH DTO -- Tum alanlar nullable
public class ProductPatchDTO {
    private String name;         // null = gonderilmedi
    private BigDecimal price;    // null = gonderilmedi
    private String description;
    private String category;
    // Getter/Setter...
}

// Servis -- null olan alanlara dokunma
@Transactional
public void patchProductWithDto(Long id, ProductPatchDTO dto) {
    Product product = repository.findById(id).orElseThrow();

    if (dto.getName() != null) {
        product.setName(dto.getName());
    }
    if (dto.getPrice() != null) {
        product.setPrice(dto.getPrice());
    }
    if (dto.getDescription() != null) {
        product.setDescription(dto.getDescription());
    }
    if (dto.getCategory() != null) {
        product.setCategory(dto.getCategory());
    }

    // KISITLAMA: Bu yaklasimda client bir alani bilerek null yapamaz!
    // Cunku null = "gonderilmedi" ile null = "temizle" ayrimi yapilamiyor.
}

Uyari: DTO + Tip Guvenligi yonteminde onemli bir kisitlama vardir: Client bir alani bilerek null yapamaz. Cunku null = "gonderilmedi" ile null = "temizle" ayrimi yapilamaz. Bu sorunu cozmek icin JsonNullable kullanin (Bolum 15.3).

Bilgi Kontrolu

1. PUT ve PATCH arasindaki temel fark nedir?

✓ Dogru! PUT "gonderdigimle tamamen degistir" anlamina gelirken, PATCH "sadece gonderdigim alanlari guncelle" anlamina gelir.
✗ Yanlis. Dogru cevap B: PUT tum alanlari ezerek yazar (gonderilmeyen alanlar null olur), PATCH ise sadece gonderilen alanlari gunceller ve diger alanlara dokunmaz.

2. Controller'da Entity yerine neden DTO kullanilmalidir?

✓ Dogru! Entity'yi dogrudan JSON'a baglamak, kullanicinin ID, createdAt gibi hassas alanlari degistirmesine olanak tanir. DTO kullanarak hangi alanlarin guncellenebilecegini kontrol edersiniz.
✗ Yanlis. Dogru cevap C: Entity'yi dogrudan JSON'a baglamak guvenlik acigi olusturur. Kullanici ID, createdAt gibi alanlari degistirebilir ve Entity DETACHED durumda olur.

3. Managed Entity guncellendiginde neden save() cagrisina gerek yoktur?

✓ Dogru! Managed durumundaki Entity'ler icin Hibernate'in Dirty Checking mekanizmasi, transaction sonunda degisiklikleri otomatik olarak veritabanina yansitir.
✗ Yanlis. Dogru cevap A: Dirty Checking mekanizmasi, Persistence Context'teki managed Entity'lerin degisikliklerini otomatik olarak algilar ve transaction commit sirasinda UPDATE sorgusu olusturur.

15 DTO Mapping Araclari: MapStruct, Jackson, JsonNullable

Neden DTO Mapping Araci?

30-40 alanli Entity'lerde elle set/get yazmak surdurulebilir degildir. Sektor bu boilerplate'i MapStruct, Jackson ve JsonNullable gibi araclarla cozer.

15.1 MapStruct (Sektor Standardi)

MapStruct derleme zamaninda (compile-time) calisir, reflection kullanmaz, cok hizlidir.

Maven Bagimliligi

<dependency>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct</artifactId>
    <version>1.5.5.Final</version>
</dependency>
<dependency>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct-processor</artifactId>
    <version>1.5.5.Final</version>
    <scope>provided</scope>
</dependency>

Mapper Arayuzu

import org.mapstruct.*;

@Mapper(componentModel = "spring")
public interface ProductMapper {

    // Entity -> DTO donusumu (Okuma icin)
    ProductResponseDTO toResponseDto(Product entity);

    // DTO -> Entity (Yeni kayit olusturma icin)
    Product toEntity(ProductCreateDTO dto);

    // PUT: DTO'daki HER SEYI Entity'nin uzerine yazar (null dahil)
    void updateFromPutDto(ProductPutDTO dto, @MappingTarget Product entity);

    // PATCH: DTO'daki null olan alanlari ATLA, sadece dolu olanlari guncelle
    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
    void updateFromPatchDto(ProductPatchDTO dto, @MappingTarget Product entity);
}

@MappingTarget Anotasyonu

@MappingTarget ile isaretlenen parametre, MapStruct'in uzerine yazacagi hedef nesnedir. Bu sayede mevcut bir Entity'yi DTO'daki degerlerle guncelleyebilirsiniz.

NullValuePropertyMappingStrategy.IGNORE

PATCH islemlerinde @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE) kullanarak, DTO'daki null olan alanlar Entity'ye yazilmaz. Boylece sadece gonderilen alanlar guncellenir.

Servis Kullanimi (Boilerplate Sifir!)

@Service
public class ProductService {

    @Autowired private ProductRepository repository;
    @Autowired private ProductMapper mapper;

    // PUT
    @Transactional
    public void putProduct(Long id, ProductPutDTO dto) {
        Product product = repository.findById(id).orElseThrow();
        mapper.updateFromPutDto(dto, product);
        // Tum alanlar ezildi. Dirty Checking UPDATE atacak.
    }

    // PATCH
    @Transactional
    public void patchProduct(Long id, ProductPatchDTO dto) {
        Product product = repository.findById(id).orElseThrow();
        mapper.updateFromPatchDto(dto, product);
        // Sadece null OLMAYAN alanlar guncellendi.
    }
}

15.2 Jackson readerForUpdating (DTO'suz PATCH)

@Service
public class ProductService {

    @Autowired private ProductRepository repository;
    @Autowired private ObjectMapper objectMapper;  // Spring Boot otomatik saglar

    @Transactional
    public void patchProduct(Long id, JsonNode patchJson) throws IOException {
        Product product = repository.findById(id).orElseThrow();

        // JSON'daki alanlari dogrudan Entity'nin uzerine "yamala"
        objectMapper.readerForUpdating(product).readValue(patchJson);

        // Dirty Checking devreye girer, UPDATE atar.
    }
}

// Controller
@PatchMapping("/{id}")
public ResponseEntity<Void> patch(@PathVariable Long id,
                                  @RequestBody JsonNode patchJson) throws IOException {
    productService.patchProduct(id, patchJson);
    return ResponseEntity.noContent().build();
}

Guvenlik Uyarisi: Bu yontemde Entity dogrudan JSON'a acilir. @JsonIgnore veya @JsonProperty(access = READ_ONLY) ile hassas alanlari (id, createdAt, role) koruyun.

15.3 JsonNullable (Gercek RESTful PATCH)

JsonNullable Nedir?

"Gonderilmedi" ile "bilerek null yapti" ayrimini yapar. Uc durumu ifade eder: undefined() (JSON'da alan yok -- dokunma), of("deger") (JSON'da deger var -- guncelle), of(null) (JSON'da alan var ama degeri null -- null yap).

Maven Bagimliligi

<dependency>
    <groupId>org.openapitools</groupId>
    <artifactId>jackson-databind-nullable</artifactId>
    <version>0.2.6</version>
</dependency>

DTO Tanimi

import org.openapitools.jackson.nullable.JsonNullable;

public class ProductPatchDTO {

    // JsonNullable 3 durumu ifade eder:
    // 1. undefined() -> JSON'da bu alan YOK (dokunma)
    // 2. of("deger")  -> JSON'da deger var (guncelle)
    // 3. of(null)     -> JSON'da alan var ama degeri null (null yap)
    private JsonNullable<String> name = JsonNullable.undefined();
    private JsonNullable<BigDecimal> price = JsonNullable.undefined();
    private JsonNullable<String> description = JsonNullable.undefined();

    // Getter/Setter...
}

Servis

@Transactional
public void patchProduct(Long id, ProductPatchDTO dto) {
    Product product = repository.findById(id).orElseThrow();

    // isPresent(): JSON'da bu anahtar var mi? (null olsa bile true doner)
    if (dto.getName().isPresent()) {
        product.setName(dto.getName().get());  // null da olabilir
    }
    if (dto.getPrice().isPresent()) {
        product.setPrice(dto.getPrice().get());
    }
    if (dto.getDescription().isPresent()) {
        product.setDescription(dto.getDescription().get());
    }
}

JSON Ornekleri

// Ismi guncelle, aciklamayi null yap, fiyata dokunma
{ "name": "Yeni Isim", "description": null }
// -> name.isPresent() = true, name.get() = "Yeni Isim"
// -> description.isPresent() = true, description.get() = null
// -> price.isPresent() = false (dokunulmaz)

Ipucu: JsonNullable, gercek RESTful PATCH semantigi icin en dogru cozumdur. isPresent() metodu JSON'da anahtarin var olup olmadigini kontrol eder -- deger null olsa bile true doner. Bu sayede "gonderilmedi" ile "bilerek null yapti" ayrimi net bir sekilde yapilabilir.

Bilgi Kontrolu

1. MapStruct'in en buyuk avantaji nedir?

✓ Dogru! MapStruct compile-time'da kod uretir, reflection kullanmaz ve bu sayede calisma zamaninda cok hizli calisir.
✗ Yanlis. Dogru cevap A: MapStruct derleme zamaninda (compile-time) calisir, reflection kullanmaz ve bu sayede cok hizli ve tip guvenli mapping saglar.

2. JsonNullable'in cozdugu temel problem nedir?

✓ Dogru! JsonNullable uc durumu ayirt eder: undefined (alan gonderilmedi), of(deger) (deger var) ve of(null) (bilerek null yapildi).
✗ Yanlis. Dogru cevap C: JsonNullable'in temel amaci PATCH islemlerinde "JSON'da alan yok" ile "JSON'da alan var ama degeri null" durumlarini birbirinden ayirmaktir.

3. Jackson readerForUpdating yontemi kullanildiginda hangi guvenlik onlemi alinmalidir?

✓ Dogru! Entity dogrudan JSON'a acildigi icin id, createdAt, role gibi hassas alanlari @JsonIgnore veya @JsonProperty(access = READ_ONLY) ile korumak gerekir.
✗ Yanlis. Dogru cevap B: Entity dogrudan JSON'a acildigi icin @JsonIgnore veya @JsonProperty(access = READ_ONLY) ile hassas alanlarin (id, createdAt, role) degistirilmesi engellenmelidir.

16 Sık Yapılan Hatalar ve Çözümleri

16.1 N+1 Güncelleme Problemi

❌ Yanlış

Her kayıt için ayrı UPDATE sorgusu (1000 kayıt = 1000 sorgu)

@Transactional
public void giveRaise(String department) {
    List<Employee> employees = employeeRepository.findByDepartment(department);
    for (Employee emp : employees) {
        emp.setSalary(emp.getSalary().multiply(new BigDecimal("1.10")));
        // Hibernate her biri icin ayri UPDATE atar!
    }
}

✅ Doğru

Tek sorgu ile toplu güncelleme

@Modifying
@Query("UPDATE Employee e SET e.salary = e.salary * 1.10 WHERE e.department = :dept")
int bulkRaise(@Param("dept") String department);
// 1000 kayit olsa bile tek bir UPDATE sorgusu calisir.

16.2 N+1 SELECT Problemi (Lazy Loading Tuzağı)

❌ Yanlış

Entity'de varsayılan Lazy Loading ile döngüde erişim

@OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
private List<OrderItem> items;

// Kodda:
List<Order> orders = orderRepository.findAll();  // 1 SELECT
for (Order order : orders) {
    order.getItems().size();  // Her siparis icin 1 SELECT daha!
    // 100 siparis = 1 + 100 = 101 sorgu!
}

✅ Doğru — Çözüm 1: JOIN FETCH

Tek sorguda ilişkili verileri getir

@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.status = :status")
List<Order> findWithItems(@Param("status") String status);

✅ Doğru — Çözüm 2: @EntityGraph

@EntityGraph(attributePaths = {"items"})
List<Order> findByStatus(String status);

✅ Doğru — Çözüm 3: DTO Projection (En iyisi)

Entity değil, sadece ihtiyacın olanı çek

@Query("SELECT new com.example.dto.OrderSummary(o.id, o.status, SIZE(o.items)) " +
       "FROM Order o WHERE o.status = :status")
List<OrderSummary> findSummariesByStatus(@Param("status") String status);

16.3 Büyük Veri Setlerini Managed Olarak Çekmek (Bellek Tükenmesi)

❌ Yanlış

50.000 kaydı Managed olarak çekmek

@Transactional
public void processAllProducts() {
    List<Product> all = productRepository.findAll();
    // 50.000 Entity + 50.000 Snapshot = 100.000 nesne RAM'de!
    // OutOfMemoryError riski!
}

✅ Doğru — Çözüm 1: readOnly

Salt okunur işlemler için readOnly kullan

@Transactional(readOnly = true)
public List<Product> getAllProducts() {
    return productRepository.findAll();
    // readOnly = true: Snapshot alinmaz -> Bellek yariya iner
}

✅ Doğru — Çözüm 2: DTO Projection (En iyi çözüm)

@Query("SELECT new com.example.dto.ProductSummary(p.id, p.name, p.price) FROM Product p")
List<ProductSummary> findAllSummaries();
// DTO'lar MANAGED olmaz. Snapshot alinmaz. Minimum bellek kullanir.

✅ Doğru — Çözüm 3: Sayfalama (Pagination)

Page<Product> products = productRepository.findAll(PageRequest.of(0, 50));

16.4 Detached Entity'yi Güncellemeye Çalışmak

❌ Yanlış

Controller'dan gelen Entity DETACHED durumda

@PostMapping("/update")
public void update(@RequestBody Product product) {
    product.setPrice(new BigDecimal("999"));
    // Bu degisiklik DB'ye YANSIMAZ! product Detached.
    // productRepository.save(product) yazsan bile merge() calisir ki
    // bu da guvenlik acigi olabilir (client tum alanlari kontrol eder).
}

✅ Doğru

Fetch-Update deseni kullan

@Transactional
public void updatePrice(Long id, BigDecimal newPrice) {
    Product product = productRepository.findById(id).orElseThrow();  // MANAGED
    product.setPrice(newPrice);
    // Dirty Checking -> UPDATE
}

Bilgi Kontrolü

1. N+1 güncelleme probleminin çözümü nedir?

✓ Doğru! Tek bir bulk UPDATE sorgusu ile 1000 kayıt bile tek sorguda güncellenir.
✗ Yanlış. Doğru cevap: B) @Modifying @Query ile tek bir bulk UPDATE sorgusu yazılmalıdır.

2. N+1 SELECT problemini çözmek için en iyi yöntem hangisidir?

✓ Doğru! DTO Projection en iyi çözümdür — Entity değil, sadece ihtiyacınız olan alanları çeker.
✗ Yanlış. Doğru cevap: C) DTO Projection — en iyi çözüm olarak önerilmektedir.

3. Detached bir entity üzerinde yapılan değişiklik neden veritabanına yansımaz?

✓ Doğru! Detached entity, Persistence Context dışındadır ve Dirty Checking tarafından izlenmez.
✗ Yanlış. Doğru cevap: A) Detached entity Persistence Context dışında olduğu için Dirty Checking çalışmaz.

17.3 Tam Calisan Ornek: Product CRUD API

DTO'lar

public record ProductCreateDTO(
    @NotBlank String name,
    @NotNull BigDecimal price,
    String description,
    @NotBlank String category
) {}

public record ProductResponseDTO(
    Long id,
    String name,
    BigDecimal price,
    String description,
    String category,
    LocalDateTime createdAt
) {}

public record ProductPutDTO(
    @NotBlank String name,
    @NotNull BigDecimal price,
    String description,
    @NotBlank String category
) {}

Controller

@RestController
@RequestMapping("/api/products")
public class ProductController {

    @Autowired private ProductService productService;

    @PostMapping
    public ResponseEntity<ProductResponseDTO> create(@Valid @RequestBody ProductCreateDTO dto) {
        ProductResponseDTO created = productService.create(dto);
        return ResponseEntity.status(HttpStatus.CREATED).body(created);
    }

    @GetMapping("/{id}")
    public ProductResponseDTO getById(@PathVariable Long id) {
        return productService.getById(id);
    }

    @PutMapping("/{id}")
    public ResponseEntity<Void> put(@PathVariable Long id,
                                    @Valid @RequestBody ProductPutDTO dto) {
        productService.putProduct(id, dto);
        return ResponseEntity.noContent().build();
    }

    @PatchMapping("/{id}")
    public ResponseEntity<Void> patch(@PathVariable Long id,
                                      @RequestBody Map<String, Object> updates) {
        productService.patchProduct(id, updates);
        return ResponseEntity.noContent().build();
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        productService.deleteProduct(id);
        return ResponseEntity.noContent().build();
    }
}

Servis

@Service
public class ProductService {

    @Autowired private ProductRepository repository;
    @Autowired private ProductMapper mapper;

    @Transactional
    public ProductResponseDTO create(ProductCreateDTO dto) {
        Product product = mapper.toEntity(dto);
        product.setCreatedAt(LocalDateTime.now());
        product.setStatus(ProductStatus.ACTIVE);
        Product saved = repository.save(product);  // save() ZORUNLU (yeni kayit)
        return mapper.toResponseDto(saved);
    }

    @Transactional(readOnly = true)
    public ProductResponseDTO getById(Long id) {
        Product product = repository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Urun bulunamadi: " + id));
        return mapper.toResponseDto(product);
    }

    @Transactional
    public void putProduct(Long id, ProductPutDTO dto) {
        Product product = repository.findById(id).orElseThrow();
        mapper.updateFromPutDto(dto, product);  // Tum alanlar ezilir
        // save() YOK -- Managed Entity
    }

    @Transactional
    public void patchProduct(Long id, Map<String, Object> updates) {
        Product product = repository.findById(id).orElseThrow();

        if (updates.containsKey("name")) {
            product.setName((String) updates.get("name"));
        }
        if (updates.containsKey("price")) {
            product.setPrice(new BigDecimal(updates.get("price").toString()));
        }
        if (updates.containsKey("description")) {
            product.setDescription((String) updates.get("description"));
        }
        if (updates.containsKey("category")) {
            product.setCategory((String) updates.get("category"));
        }
        // save() YOK -- Managed Entity
    }

    @Transactional
    public void deleteProduct(Long id) {
        Product product = repository.findById(id).orElseThrow();
        repository.delete(product);
    }
}

Altin Kural: Fetch - Update - Let Dirty Checking Handle It

Yani: Getir - Guncelle - Hibernate'e birak. Managed Entity'lerde save() cagirmak gereksizdir; Dirty Checking degisiklikleri otomatik olarak veritabanina yansitir. Ancak yeni kayit olustururken save() zorunludur.
