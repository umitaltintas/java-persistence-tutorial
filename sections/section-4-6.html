4 Hibernate — JPA'nın Motoru

Hibernate Nedir?

Hibernate, JPA spesifikasyonunu uygulayan en yaygın ORM çerçevesidir. Spring Boot kullandığında arka planda Hibernate çalışır.

4.1 Hibernate'in Yaptığı İş

# Giriş Çıkış
1 Entity sınıflarını okur SQL CREATE TABLE ifadeleri üretir (geliştirme ortamında)
2 save() çağrısını SQL INSERT INTO ifadesine çevirir
3 JPQL sorgularını Veri tabanına özel SQL'e çevirir
4 Entity değişikliklerini takip eder Otomatik UPDATE üretir (Dirty Checking)
5 İlişkileri (Relationship) yönetir Gerektiğinde JOIN sorguları üretir

4.2 EntityManager — JPA'nın Temel Arayüzü

@Service
public class ProductServiceWithEM {

    @PersistenceContext
    private EntityManager entityManager;

    @Transactional
    public void entityManagerExamples() {

        // --- persist(): Yeni kayit ekler ---
        Product newProduct = new Product();
        newProduct.setName("Mouse");
        newProduct.setPrice(new BigDecimal("29.99"));
        entityManager.persist(newProduct);
        // Bu noktada newProduct MANAGED durumuna gecti
        // Henuz DB'ye INSERT atilmamis olabilir (flush bekliyor)

        // --- find(): ID ile bulur ---
        Product found = entityManager.find(Product.class, 1L);
        // found artik MANAGED. Degisiklikler otomatik kaydedilecek.

        // --- JPQL Query ---
        List<Product> expensive = entityManager
            .createQuery("SELECT p FROM Product p WHERE p.price > :min", Product.class)
            .setParameter("min", new BigDecimal("100"))
            .getResultList();

        // --- remove(): Siler ---
        Product toDelete = entityManager.find(Product.class, 99L);
        if (toDelete != null) {
            entityManager.remove(toDelete);
            // Transaction bittiginde DELETE sorgusu calisir
        }

        // --- detach(): Persistence Context'ten cikarir ---
        entityManager.detach(found);
        // Artik found uzerindeki degisiklikler DB'ye YANSIMAZ

        // --- merge(): Detached nesneyi tekrar Managed yapar ---
        found.setName("Guncellenmis Isim");
        Product mergedProduct = entityManager.merge(found);
        // mergedProduct MANAGED, found hala DETACHED!
        // Donen nesneyi kullanmak onemli.

        // --- flush(): Bekleyen SQL'leri hemen DB'ye gonderir ---
        entityManager.flush();
        // Normalde transaction sonunda otomatik yapilir.
        // flush() sadece siralamanin onemli oldugu durumlarda kullanilir.
    }
}
İpucu: persist() çağrıldığında nesne hemen veritabanına yazılmaz. SQL, flush() yapılana veya transaction commit edilene kadar bekletilir. Bu sayede Hibernate birden fazla işlemi toplu olarak (batch) gönderebilir.

EntityManager Temel Metotları

persist() — Yeni nesneyi Managed yapar (INSERT bekletir).
find() — ID ile bulur, Managed döndürür.
remove() — Managed nesneyi siler (DELETE bekletir).
detach() — Nesneyi Persistence Context'ten çıkarır.
merge() — Detached nesneyi tekrar Managed yapar ve yeni bir Managed kopya döndürür.
flush() — Bekleyen tüm SQL'leri hemen veritabanına gönderir.

Dikkat: merge() metodu orijinal nesneyi Managed yapmaz! Yeni bir Managed kopya döndürür. Her zaman dönen nesneyi kullanın: Product merged = entityManager.merge(detachedProduct);

Bilgi Kontrolü

1. entityManager.persist(product) çağrıldığında ne olur?

✓ Doğru! persist() nesneyi Persistence Context'e ekler (MANAGED yapar) ancak SQL hemen çalıştırılmaz. INSERT sorgusu flush() veya transaction commit anında gönderilir.
✗ Yanlış. Doğru cevap B. persist() nesneyi MANAGED yapar ama SQL hemen çalışmaz; flush veya commit bekler.

2. merge() metodu hakkında hangisi doğrudur?

✓ Doğru! merge() orijinal nesneyi değiştirmez. Yeni bir MANAGED kopya oluşturur ve onu döndürür. Bu yüzden dönen değeri kullanmak çok önemlidir.
✗ Yanlış. Doğru cevap C. merge() yeni bir MANAGED kopya döndürür; orijinal nesne hala DETACHED'tır.

3. Hibernate aşağıdakilerden hangisini yapmaz?

✓ Doğru! Hibernate bir ORM çerçevesidir ve yalnızca Java nesneleri ile veritabanı arasındaki eşlemeyi yönetir. Frontend teknolojileriyle ilgilenmez.
✗ Yanlış. Doğru cevap A. Hibernate yalnızca Java-veritabanı eşlemesi yapar; frontend ile ilgisi yoktur.

5 Persistence Context ve Entity Yaşam Döngüsü

Neden Bu Konu Çok Önemli?

Bu konu, JPA (Hibernate) kullanan bir geliştiricinin "Junior"dan "Senior"a geçişindeki en kritik bilgidir. Persistence Context'i anlamadan yazılan kod, gereksiz sorgular üretir, beklenmedik hatalar verir ve performans sorunlarına yol açar.

5.1 Persistence Context Nedir?

Persistence Context, Java kodunuz ile veri tabanı arasında duran bir tampon bellek (buffer) ve değişiklik takipçisidir.

+--------------+     +-------------------------+     +--------------+
|  Java Kodu   | <-> |  Persistence Context    | <-> |  Veri Tabani |
|  (Uygulama)  |     |  (1. Seviye On Bellek)  |     |  (SQL DB)    |
+--------------+     +-------------------------+     +--------------+

Persistence Context'in İki Temel Görevi

1. Dirty Checking: Nesnelerin değişip değişmediğini takip eder. Değişen nesneler için otomatik UPDATE sorgusu üretir.
2. Kimlik Garantisi (Identity Map): Aynı ID'ye sahip veriyi aynı transaction içinde iki kez çekersen, DB'ye gitmez; hafızadaki aynı nesneyi döndürür.

5.2 Entity'nin 4 Yaşam Evresi

TRANSIENT
Yeni / Geçici

new Product() ile oluşturulmuş, henüz Persistence Context'e eklenmemiş nesne. Veritabanı bu nesneden habersizdir. Garbage Collector istediği zaman silebilir.

persist() / save() →
MANAGED
Yönetilen

Persistence Context tarafından takip edilen nesne. persist(), find() veya JPQL sorgusuyla bu duruma geçer. Üzerinde yapılan tüm değişiklikler transaction sonunda otomatik olarak veritabanına yansır. Tüm sihir burada gerçekleşir!

detach() / transaction kapanır ↓
remove() ↓
DETACHED
Kopmuş

Daha önce Managed olan ama artık Persistence Context'te olmayan nesne. detach() çağrısı veya transaction kapanmasıyla bu duruma geçer. Üzerindeki değişiklikler veritabanına yansımaz. merge() ile tekrar Managed yapılabilir.

merge() ↑
REMOVED
Silinmiş

remove() ile silinmek üzere işaretlenmiş nesne. Transaction commit edildiğinde veritabanından DELETE sorgusu ile kaldırılır.

    new Product()          persist() / save()
  +-------------+       +-----------------+
  |  TRANSIENT  | ----> |    MANAGED      | <---- find() / query
  |  (Yeni)     |       |  (Yonetilen)    |
  +-------------+       +--------+--------+
                                 |
                    +------------+------------+
                    | detach()   | remove()   |
                    v            v            |
              +-----------+  +----------+    |
              | DETACHED  |  | REMOVED  |    |
              | (Kopmus)  |  | (Silinm.)|    |
              +-----+-----+  +----------+    |
                    |                         |
                    +-- merge() --------------+

A. TRANSIENT (Yeni / Geçici)

Product product = new Product();
product.setName("Laptop");
// Su an TRANSIENT. DB bilmiyor, Hibernate bilmiyor.
// Garbage Collector bu nesneyi istedigi zaman silebilir.

B. MANAGED (Yönetilen) — İşte Tüm Sihir Burada

@Transactional
public void managedExample() {
    // Yol 1: persist ile Managed yapma
    Product newProduct = new Product();
    newProduct.setName("Tablet");
    entityManager.persist(newProduct);  // Artik MANAGED

    // Yol 2: find ile Managed yapma (en yaygin yol)
    Product existing = entityManager.find(Product.class, 1L);  // MANAGED

    // MANAGED nesne uzerindeki degisiklikler OTOMATIK kaydedilir:
    existing.setPrice(new BigDecimal("999.99"));
    // Burada save() veya update() cagrisi YOK!
    // Transaction bittiginde Hibernate bunu kendi halleder.
}
Önemli: MANAGED bir nesne üzerinde yapılan değişiklikler için save() veya update() çağrısı gerekmez. Transaction bittiğinde Hibernate, değişiklikleri otomatik olarak veritabanına yazar. Bu mekanizmaya Dirty Checking denir.

C. DETACHED (Kopmuş)

@Transactional
public Product getProduct(Long id) {
    return productRepository.findById(id).orElseThrow();
    // Metot bitip @Transactional kapaninca, donen nesne DETACHED olur.
}

// Controller'da:
public void updatePrice() {
    Product p = productService.getProduct(1L);
    p.setPrice(new BigDecimal("500"));
    // BU DEGISIKLIK VERI TABANINA YANSIMAZ!
    // Cunku product artik DETACHED. Persistence Context kapandi.
}
Sık Yapılan Hata: @Transactional metot bittiğinde dönen entity nesnesi DETACHED olur. Controller katmanında bu nesne üzerinde yapılan değişiklikler veritabanına yansımaz. Bu, özellikle yeni başlayanların en çok yaptığı hatalardan biridir.

D. REMOVED (Silinmiş)

@Transactional
public void deleteProduct(Long id) {
    Product product = entityManager.find(Product.class, id);  // MANAGED
    entityManager.remove(product);  // REMOVED
    // Transaction bittiginde DELETE sorgusu calisir.
}

5.3 Persistence Context'in Kimlik Garantisi

Identity Map Deseni

Persistence Context, aynı transaction içinde aynı ID'ye sahip entity için her zaman aynı Java nesne referansını döndürür. Bu, sadece eşit değer (equals) değil, aynı referans (==) anlamına gelir. İkinci sorgu veritabanına gitmez; ön bellekten (cache) döner.

@Transactional
public void identityMapDemo() {
    // Ilk cagri: DB'ye SELECT sorgusu gider
    Product p1 = productRepository.findById(1L).get();

    // Ikinci cagri: DB'ye sorgu GITMEZ! Persistence Context'ten doner.
    Product p2 = productRepository.findById(1L).get();

    // Ayni nesne mi?
    System.out.println(p1 == p2);  // true -- Sadece deger degil, referans da ayni!
}
Performans Avantajı: Identity Map sayesinde aynı entity'yi bir transaction içinde kaç kez sorgularsanız sorgulayın, veritabanına yalnızca bir kez gidilir. Bu, özellikle karmaşık iş mantığı olan servislerde önemli bir performans kazancı sağlar.

Bilgi Kontrolü

1. Aşağıdaki kodda product nesnesi hangi durumdadır?
Product product = new Product(); product.setName("Test");

✓ Doğru! new ile oluşturulan ve henüz persist() veya save() ile Persistence Context'e eklenmemiş nesne TRANSIENT durumdadır. Veritabanı ve Hibernate bu nesneden habersizdir.
✗ Yanlış. Doğru cevap C (TRANSIENT). new ile oluşturulan nesne henüz Persistence Context'e eklenmediği için TRANSIENT durumdadır.

2. @Transactional bir metottan dönen entity nesnesi, metot dışında hangi duruma geçer?

✓ Doğru! @Transactional metot bittiğinde Persistence Context kapanır ve tüm Managed nesneler DETACHED durumuna geçer. Bu nesneler üzerindeki değişiklikler artık veritabanına otomatik olarak yansımaz.
✗ Yanlış. Doğru cevap A (DETACHED). Transaction kapandığında Persistence Context da kapanır ve nesneler DETACHED olur.

3. Aynı transaction içinde findById(1L) iki kez çağrılırsa ne olur?

✓ Doğru! Persistence Context'in Identity Map özelliği sayesinde aynı ID ile yapılan ikinci sorgu veritabanına gitmez. Ön bellekteki aynı nesne referansı döndürülür.
✗ Yanlış. Doğru cevap B. Persistence Context, Identity Map deseni ile aynı ID'li entity için tek bir SELECT yapar ve aynı referansı döndürür.

6 Dirty Checking Mekanizması

Dirty Checking Nedir?

Dirty Checking, Hibernate'in MANAGED durumdaki entity nesnelerinin değişip değişmediğini otomatik olarak takip etmesi ve değişen ("kirli" — dirty) nesneler için otomatik UPDATE sorgusu üretmesidir. Bu mekanizma sayesinde geliştirici save() veya update() çağrısı yapmak zorunda kalmaz.

6.1 Nasıl Çalışır?

Adım Açıklama
1 findById(1) çağrılır → Hibernate DB'den veriyi çeker, Java nesnesi oluşturur
2 Bu nesneyi Persistence Context'e koyar
3 Gizli İşlem: Nesnenin o anki halinin bir kopyasını (snapshot) alır
4 Transaction biterken, mevcut nesneyi snapshot ile karşılaştırır
5 Farklılık varsa (nesne "kirli" — dirty), otomatik UPDATE SQL'i oluşturur ve çalıştırır 
  findById(1L) cagrilir
        |
        v
  +----------------------------------------------+
  |         Persistence Context                   |
  |                                               |
  |  Nesne:    { name: "Laptop", price: 1500 }    |
  |  Snapshot: { name: "Laptop", price: 1000 }    |
  |                                               |
  |  Karsilastirma: price FARKLI!                 |
  |  => UPDATE products SET price=1500            |
  |     WHERE id=1                                |
  +----------------------------------------------+

6.2 Kod Üzerinde Gösterim

@Service
public class ProductService {

    @Autowired
    private ProductRepository productRepository;

    @Transactional
    public void updateProductPrice(Long id, BigDecimal newPrice) {

        // Adim 1: DB'den oku -> Nesne MANAGED oldu
        // Adim 2: Hibernate arka planda snapshot aldi:
        //         snapshot = { name: "Laptop", price: 1000.00, ... }
        Product product = productRepository.findById(id).orElseThrow();

        // Adim 3: Sadece Java nesnesinin degerini degistir
        product.setPrice(newPrice);

        // Adim 4: Metot biter, @Transactional kapanir
        // Hibernate karsilastirir:
        //   mevcut  = { name: "Laptop", price: 1500.00, ... }
        //   snapshot = { name: "Laptop", price: 1000.00, ... }
        //   FARK VAR! -> UPDATE products SET price = 1500.00 WHERE id = ?

        // DIKKAT: productRepository.save(product) YAZILMAZ!
        // Cunku nesne zaten MANAGED. save() gereksiz islem yapar.
    }
}
Neden save() Gereksiz? Nesne zaten MANAGED durumda olduğu için Hibernate değişiklikleri snapshot mekanizması ile otomatik algılar. Ekstra save() çağrısı yapmak, Spring Data'nın arka planda gereksiz kontroller yapmasına neden olur (ID var mı? → merge() çağırır → ama zaten Managed...) ve boşuna CPU döngüsü harcanır.

6.3 save() Çağrısının Gereksiz Olduğu Durumlar

// YANLIS -- Gereksiz save() cagrisi
@Transactional
public void badExample(Long id) {
    Product product = productRepository.findById(id).orElseThrow();
    product.setName("Yeni Isim");
    productRepository.save(product);  // GEREKSIZ! Zaten Managed.
    // Spring Data arka planda:
    // 1. "Bu nesnenin ID'si var mi?" -> Evet
    // 2. "O zaman merge() cagirayim" -> Ama zaten Managed...
    // 3. Bosuna CPU dongusu harcandi.
}
// DOGRU -- Temiz ve verimli
@Transactional
public void goodExample(Long id) {
    Product product = productRepository.findById(id).orElseThrow();
    product.setName("Yeni Isim");
    // Bitti. Hibernate halleder.
}
Kural: Eğer bir entity'yi findById(), find() veya herhangi bir sorgu ile aldıysanız ve @Transactional bir metot içindeyseniz, değişikliklerden sonra save() çağırmayın. Bu gereksiz bir işlemdir ve kodun okunabilirliğini azaltır. save() yalnızca yeni (TRANSIENT) nesneler için kullanılmalıdır.

Özet: Dirty Checking İş Akışı

1. Entity, find() veya sorgu ile okunur → MANAGED olur.
2. Hibernate, nesnenin anlık görüntüsünü (snapshot) kaydeder.
3. Geliştirici, setter metotları ile nesneyi değiştirir.
4. Transaction commit olurken Hibernate, mevcut nesneyi snapshot ile karşılaştırır.
5. Fark varsa otomatik UPDATE SQL'i üretilir ve çalıştırılır.
6. Fark yoksa hiçbir SQL üretilmez — performans kazanımı!
