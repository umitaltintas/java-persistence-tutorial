21 Entity-First vs Table-First Tasarım

Thorben Janssen'in "Unpopular Opinion"i

Thorben: "Önce entitymı yoksa tablo mu tasarlanmalı? Fark etmez -- bu benim unpopular opinion'im." Günümüzde çoğu uygulamada 1 uygulama + 1 veritabanı var ve ikisinin birlikte çalışması lazım. Hangi taraftan başlarsan başla, sonuçta her iki tarafı (Java + DB) birbirine uyumlu hale getirmek için ayarlamalar yapılacak.

21.1 İki Yaklaşım

Özellik Entity-First Table-First
Başlangıç noktası Java sınıflarını yaz, Hibernate tablo üretsin Öncelikle tablo yapısını tasarla, sonra Entity yaz
Tablo oluşturma ddl-auto=update ile otomatik SQL migration scriptleri ile manuel
Avantaj Hızlı prototipleme, Java-odaklı düşünme DB performansı ve index kontrolü, DBA uyumlu
Dezavantaj Üretilen tablolar optimize olmayabilir Daha fazla ön tasarım gerektirir
Uygun ortam Prototip, hackathon, küçük projeler Production, büyük projeler, DBA olan takımlar

21.2 Entity-First Yaklaşımı

Java sınıflarından başlayarak Hibernate'in ddl-auto özelliği ile tablo üretimine izin verirsiniz. Bu yaklaşım sadece geliştirme ortamında kullanılmalıdır.

// Entity-First: Öncelikle Java sınıfını yaz
@Entity
@Table(name = "products")  // Tablo adını açıkça belirt
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "product_name", nullable = false, length = 200)
    private String name;

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    // Getter/Setter...
}
# application.yml -- SADECE dev ortamında!
spring:
  jpa:
    hibernate:
      ddl-auto: update  # Hibernate tablo yapısını Entity'ye göre günceller
    show-sql: true

Uyarı: ddl-auto: update veya create kesinlikle production ortamında KULLANILMAMALIDIR! Production'da her zaman validate kullanın ve şema değişikliklerini Flyway/Liquibase ile yönetin.

21.3 Table-First Yaklaşımı

Öncelikle veritabanı tablosunu tasarlayıp oluşturursunuz, sonra Entity'leri ona göre yazarsınız.

-- Table-First: Öncelikle SQL ile tablo oluştur
CREATE TABLE products (
    id          BIGSERIAL PRIMARY KEY,
    product_name VARCHAR(200) NOT NULL,
    price       DECIMAL(10, 2),
    description TEXT,
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP,
    category_id BIGINT REFERENCES categories(id)
);

CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_name ON products(product_name);
// Table-First: Mevcut tabloya göre Entity yaz
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "product_name", nullable = false, length = 200)
    private String name;  // Java'da "name", DB'de "product_name"

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;

    // Getter/Setter...
}

DBA'ların görüşü: "Tabloyu biz tasarlarız." Thorben'in cevabı: "Her iki tarafta da ayarlama gerekecek." Önemli olan, sonuçta Entity ve tablo yapısının birbirine uyumlu olması ve performansın optimize edilmesidir.

21.4 Entity Mapping Best Practices

@Entity
@Table(name = "orders")  // Her zaman @Table ile tablo adını belirt
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // @Column ile sütun adını, kısıtlamaları ve tipi belirt
    @Column(name = "order_number", nullable = false, unique = true, length = 50)
    private String orderNumber;

    @Column(name = "total_amount", precision = 12, scale = 2, nullable = false)
    private BigDecimal totalAmount;

    @Enumerated(EnumType.STRING)  // Ordinal KULLANMA! String olarak kaydet
    @Column(name = "status", length = 20)
    private OrderStatus status;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    // Getter/Setter...
}

Naming Strategy

Spring Boot varsayılan olarak SpringPhysicalNamingStrategy kullanır. Bu strateji camelCase alan adlarını snake_case'e çevirir: orderNumber → order_number. Eğer DB'deki sütun adı farklı ise @Column(name = "...") ile açıkça belirtmelisiniz.

21.5 Kesinlikle YAPILMAMASI Gereken

Kritik Uyarı: Bir tarafı tasarlayıp diğer tarafı otomatik generate edip olduğu gibi bırakmak en büyük hatadır! Örneğin:

  • Entity yazıp ddl-auto=update ile tablo üretmek ve o tablonun yapısını hiç incelememek
  • DB tablosunu tasarlayıp Entity'yi otomatik generate edip mapping ayarlamaları yapmamak
  • Index, constraint ve performans optimizasyonlarını göz ardı etmek

Her iki durumda da: Sonuçta Entity ve tablo yapısını birbirine uyumlu hale getirmek için inceleme ve ayarlamalar YAPILMALIDIR.

22 Criteria API ve Alternatifler

JPA Criteria API Nedir?

Criteria API, JPA'nın programatik (kod ile) sorgu oluşturma mekanizmasıdır. SQL veya JPQL yazmak yerine Java nesneleri kullanarak sorgu oluşturursunuz. En büyük avantajı: Sorguların dinamik olarak (runtime'da) oluşturulabilmesidir.

22.1 Thorben'in Görüşü

Thorben Janssen: "Teknik olarak iyi bir API ama yazdığım SQL'den çok farklı görünüyor." Problem: 30-40 satır kod, okunaksız, değişken isimleri ve yorumlar olmadan anlaşılmaz. Statik sorgular için JPQL veya @Query her zaman daha okunaklıdır.

22.2 Criteria API ile Basit Bir Sorgu

// Criteria API: Fiyatı 100'den büyük ürünleri getir
@Repository
public class ProductCriteriaRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> findExpensiveProducts(BigDecimal minPrice) {
        // 1. CriteriaBuilder al
        CriteriaBuilder cb = em.getCriteriaBuilder();

        // 2. CriteriaQuery oluştür (dönüş tipi: Product)
        CriteriaQuery<Product> cq = cb.createQuery(Product.class);

        // 3. Root tanımla (FROM kısmı)
        Root<Product> root = cq.from(Product.class);

        // 4. WHERE koşulunu ekle
        cq.select(root)
          .where(cb.greaterThan(root.get("price"), minPrice));

        // 5. Sorguyu çalıştır
        return em.createQuery(cq).getResultList();
    }
}

22.3 Aynı Sorgunun JPQL Karşılığı

Criteria API (8+ satır) JPQL (1 satır) 
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Product> cq = cb.createQuery(Product.class);
Root<Product> root = cq.from(Product.class);
cq.select(root)
  .where(cb.greaterThan(root.get("price"), minPrice));
return em.createQuery(cq).getResultList();
 
@Query("SELECT p FROM Product p WHERE p.price > :minPrice")
List<Product> findExpensive(@Param("minPrice") BigDecimal minPrice);

Kural: Eğer sorgunuz statik ise (yani WHERE koşulları sabit), her zaman JPQL veya @Query kullanın. Criteria API sadece dinamik sorgular için değer kazanır.

22.4 Eski Hibernate Criteria vs JPA Criteria

Tarihçe

Eski Hibernate Criteria API (session.createCriteria()) daha basit bir API'ydi ancak JPA standardı değildi ve Hibernate 5+ ile deprecated oldu. JPA Criteria API standart olmasına rağmen daha karmaşık ve okunaksızdır. Hibernate 6/7 ile yeni "Restrictions" yaklaşımı bu sorunu hafifletmeye çalışmaktadır.

22.5 Dinamik Filtre Örneği -- Criteria API'nin Parladığı Yer

Kullanıcı bir arama formunda farklı filtreleri seçebilir: isim, kategori, fiyat aralığı... Hangi filtrelerin geleceği runtime'da belli olur. Bu senaryoda Criteria API çok değerlidir.

@Repository
public class ProductSearchRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> searchProducts(String name, String category,
                                         BigDecimal minPrice, BigDecimal maxPrice) {
        CriteriaBuilder cb = em.getCriteriaBuilder();
        CriteriaQuery<Product> cq = cb.createQuery(Product.class);
        Root<Product> root = cq.from(Product.class);

        // Dinamik olarak koşulları birleştir
        List<Predicate> predicates = new ArrayList<>();

        if (name != null && !name.isBlank()) {
            predicates.add(cb.like(cb.lower(root.get("name")),
                           "%" + name.toLowerCase() + "%"));
        }

        if (category != null && !category.isBlank()) {
            predicates.add(cb.equal(root.get("category"), category));
        }

        if (minPrice != null) {
            predicates.add(cb.greaterThanOrEqualTo(root.get("price"), minPrice));
        }

        if (maxPrice != null) {
            predicates.add(cb.lessThanOrEqualTo(root.get("price"), maxPrice));
        }

        // Tüm koşulları AND ile birleştir
        cq.select(root)
          .where(cb.and(predicates.toArray(new Predicate[0])));

        return em.createQuery(cq).getResultList();
    }
}

Neden Criteria API Burada Parlıyor?

Bu sorguyu JPQL ile yazmak için string birleştirme ("SELECT ... " + (name != null ? "AND ..." : "")) gerekir ki bu hem okunaksız hem de SQL injection risklidir. Criteria API, koşulları programatik olarak ve tip güvenli şekilde oluşturmanızı sağlar.

22.6 Spring Data JPA Specification Pattern

Spring Data JPA, Criteria API'yi saran daha temiz bir yaklaşım sunar: Specification pattern.

// 1. Repository: JpaSpecificationExecutor'i extend et
public interface ProductRepository extends JpaRepository<Product, Long>,
                                           JpaSpecificationExecutor<Product> {
}

// 2. Specification sınıfı oluştur
public class ProductSpecifications {

    public static Specification<Product> hasName(String name) {
        return (root, query, cb) ->
            name == null ? null : cb.like(cb.lower(root.get("name")),
                                         "%" + name.toLowerCase() + "%");
    }

    public static Specification<Product> hasCategory(String category) {
        return (root, query, cb) ->
            category == null ? null : cb.equal(root.get("category"), category);
    }

    public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min != null && max != null) {
                return cb.between(root.get("price"), min, max);
            } else if (min != null) {
                return cb.greaterThanOrEqualTo(root.get("price"), min);
            } else if (max != null) {
                return cb.lessThanOrEqualTo(root.get("price"), max);
            }
            return null;
        };
    }
}

// 3. Servis katmanında kullanımı
@Service
public class ProductService {

    @Autowired
    private ProductRepository repository;

    public List<Product> search(String name, String category,
                                BigDecimal minPrice, BigDecimal maxPrice) {
        Specification<Product> spec = Specification
            .where(ProductSpecifications.hasName(name))
            .and(ProductSpecifications.hasCategory(category))
            .and(ProductSpecifications.priceBetween(minPrice, maxPrice));

        return repository.findAll(spec);
    }
}

İpucu: Specification pattern, Criteria API'nin gücünü Spring Data JPA'nın kolaylığıyla birleştirir. Her filtre ayrı bir metotta tanımlanır, yeniden kullanılabilir ve test edilebilir. Yeni bir filtre eklemek için sadece yeni bir Specification metodu yazmak yeterlidir.

22.7 Hibernate 6/7: Yeni Restrictions Yaklaşımı

JPQL + Dinamik WHERE Koşulu

Hibernate 6/7, Criteria API'nin karmaşıklığını azaltmak için JPQL tabanlı dinamik sorgu oluşturma imkanı sunan yeni yaklaşımlara yönelmektedir. Temel fikir: JPQL'in okunaklılığını koruyarak WHERE koşullarını dinamik olarak eklemektir.

// Hibernate 6+ ile daha temiz dinamik sorgular
// HibernateCriteriaBuilder ek metodlar sunar
@Repository
public class ProductSearchRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> search(String name, BigDecimal minPrice) {
        HibernateCriteriaBuilder cb = (HibernateCriteriaBuilder) em.getCriteriaBuilder();
        JpaCriteriaQuery<Product> cq = cb.createQuery(Product.class);
        JpaRoot<Product> root = cq.from(Product.class);

        List<Predicate> predicates = new ArrayList<>();

        if (name != null) {
            predicates.add(cb.ilike(root.get("name"), "%" + name + "%"));
            // ilike: case-insensitive LIKE (Hibernate 6+)
        }
        if (minPrice != null) {
            predicates.add(cb.ge(root.get("price"), minPrice));
        }

        cq.select(root).where(predicates.toArray(new Predicate[0]));
        return em.createQuery(cq).getResultList();
    }
}

22.8 Ne Zaman Hangisini Kullan?

Senaryo Önerilen Yaklaşım Neden
Statik sorgu (WHERE koşulları sabit) JPQL / @Query Okunaklı, basit, SQL benzeri
Dinamik filtreler (runtime'da belirlenir) Specification pattern Criteria API'yi sarar, yeniden kullanılabilir
Çok karmaşık dinamik sorgular Criteria API (doğrudan) Tam kontrol, tip güvenliği
Karmaşık raporlar, analitik sorgular jOOQ veya Native SQL SQL'in tam gücünü kullanır

Bilgi Kontrolü

1. Criteria API'nin en büyük avantajı nedir?

✓ Doğru! Criteria API'nin en büyük avantajı, WHERE koşullarını runtime'da dinamik olarak oluşturabilmesidir. String birleştirme yerine tip güvenli nesneler kullanılır.
✗ Yanlış. Doğru cevap B: Criteria API, JPQL'den daha okunaklı değildir ve performans farkı yoktur. Gerçek avantajı dinamik sorgu oluşturma yeteneğidir.

2. Statik bir sorgu için (WHERE koşulları sabit) hangi yaklaşım tercih edilmelidir?

✓ Doğru! Statik sorgularda JPQL veya @Query her zaman daha okunaklı ve tercih edilir. Criteria API statik sorgularda gereksiz karmaşıklık ekler.
✗ Yanlış. Doğru cevap A: Thorben'in de belirttiği gibi, statik sorgular için JPQL her zaman daha okunaklıdır. Criteria API sadece dinamik sorgularda değer kazanır.

3. Spring Data JPA Specification pattern'in avantajı nedir?

✓ Doğru! Specification pattern her filtreyi bağımsız bir metot olarak tanımlar. Bu sayede filtreler yeniden kullanılabilir, test edilebilir ve .and() / .or() ile kolayca birleştirilebilir.
✗ Yanlış. Doğru cevap C: Specification pattern'in temel avantajı filtrelerin modüler, yeniden kullanılabilir ve birleştirilebilir olmasıdır. Performans farkı yoktur, arka planda Criteria API kullanır.
