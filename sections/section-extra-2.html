21 Entity-First vs Table-First Tasarim

Thorben Janssen'in "Unpopular Opinion"i

Thorben: "Once entity mi yoksa tablo mu tasarlanmali? Fark etmez -- bu benim unpopular opinion'im." Gunumuzde cogu uygulamada 1 uygulama + 1 veritabani var ve ikisinin birlikte calismasi lazim. Hangi taraftan baslarsan basla, sonucta her iki tarafi (Java + DB) birbirine uyumlu hale getirmek icin ayarlamalar yapilacak.

21.1 Iki Yaklasim

Ozellik Entity-First Table-First
Baslangic noktasi Java siniflarini yaz, Hibernate tablo uretsin Oncelikle tablo yapisini tasarla, sonra Entity yaz
Tablo olusturma ddl-auto=update ile otomatik SQL migration scriptleri ile manuel
Avantaj Hizli prototipleme, Java-odakli dusunme DB performansi ve index kontrolu, DBA uyumlu
Dezavantaj Uretilen tablolar optimize olmayabilir Daha fazla on tasarim gerektir
Uygun ortam Prototip, hackathon, kucuk projeler Production, buyuk projeler, DBA olan takimlar

21.2 Entity-First Yaklasimi

Java siniflarindan baslayarak Hibernate'in ddl-auto ozelligi ile tablo uretimine izin verirsiniz. Bu yaklasim sadece gelistirme ortaminda kullanilmalidir.

// Entity-First: Oncelikle Java sinifini yaz
@Entity
@Table(name = "products")  // Tablo adini acikca belirt
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "product_name", nullable = false, length = 200)
    private String name;

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    // Getter/Setter...
}
# application.yml -- SADECE dev ortaminda!
spring:
  jpa:
    hibernate:
      ddl-auto: update  # Hibernate tablo yapisini Entity'ye gore gunceller
    show-sql: true

Uyari: ddl-auto: update veya create kesinlikle production ortaminda KULLANILMAMALIDIR! Production'da her zaman validate kullanin ve sema degisikliklerini Flyway/Liquibase ile yonetin.

21.3 Table-First Yaklasimi

Oncelikle veritabani tablosunu tasarlayip olusturursiniz, sonra Entity'leri ona gore yazarsiniz.

-- Table-First: Oncelikle SQL ile tablo olustur
CREATE TABLE products (
    id          BIGSERIAL PRIMARY KEY,
    product_name VARCHAR(200) NOT NULL,
    price       DECIMAL(10, 2),
    description TEXT,
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP,
    category_id BIGINT REFERENCES categories(id)
);

CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_name ON products(product_name);
// Table-First: Mevcut tabloya gore Entity yaz
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "product_name", nullable = false, length = 200)
    private String name;  // Java'da "name", DB'de "product_name"

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;

    // Getter/Setter...
}

DBA'larin gorusu: "Tabloyu biz tasarlariz." Thorben'in cevabi: "Her iki tarafta da ayarlama gerekecek." Onemli olan, sonucta Entity ve tablo yapisinin birbirine uyumlu olmasi ve performansin optimize edilmesidir.

21.4 Entity Mapping Best Practices

@Entity
@Table(name = "orders")  // Her zaman @Table ile tablo adini belirt
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // @Column ile sutun adini, kisitlamalari ve tipi belirt
    @Column(name = "order_number", nullable = false, unique = true, length = 50)
    private String orderNumber;

    @Column(name = "total_amount", precision = 12, scale = 2, nullable = false)
    private BigDecimal totalAmount;

    @Enumerated(EnumType.STRING)  // Ordinal KULLANMA! String olarak kaydet
    @Column(name = "status", length = 20)
    private OrderStatus status;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    // Getter/Setter...
}

Naming Strategy

Spring Boot varsayilan olarak SpringPhysicalNamingStrategy kullanir. Bu strateji camelCase alan adlarini snake_case'e cevirir: orderNumber → order_number. Eger DB'deki sutun adi farkli ise @Column(name = "...") ile acikca belirtmelisiniz.

21.5 Kesinlikle YAPILMAMASI Gereken

Kritik Uyari: Bir tarafi tasarlayip diger tarafi otomatik generate edip oldugu gibi birakmak en buyuk hatadir! Ornegin:

  • Entity yazip ddl-auto=update ile tablo uretmek ve o tablonun yapisini hic incelememek
  • DB tablosunu tasarlayip Entity'yi otomatik generate edip mapping ayarlamalari yapmamak
  • Index, constraint ve performans optimizasyonlarini goz ardi etmek

Her iki durumda da: Sonucta Entity ve tablo yapisini birbirine uyumlu hale getirmek icin inceleme ve ayarlamalar YAPILMALIDIR.

22 Criteria API ve Alternatifler

JPA Criteria API Nedir?

Criteria API, JPA'nin programatik (kod ile) sorgu olusturma mekanizmasidir. SQL veya JPQL yazmak yerine Java nesneleri kullanarak sorgu olusturursunuz. En buyuk avantaji: Sorgularin dinamik olarak (runtime'da) olusturulabilmesidir.

22.1 Thorben'in Gorusu

Thorben Janssen: "Teknik olarak iyi bir API ama yazdigim SQL'den cok farkli gorunuyor." Problem: 30-40 satir kod, okunaksiz, degisken isimleri ve yorumlar olmadan anlasilmaz. Statik sorgular icin JPQL veya @Query her zaman daha okunaklidir.

22.2 Criteria API ile Basit Bir Sorgu

// Criteria API: Fiyati 100'den buyuk urunleri getir
@Repository
public class ProductCriteriaRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> findExpensiveProducts(BigDecimal minPrice) {
        // 1. CriteriaBuilder al
        CriteriaBuilder cb = em.getCriteriaBuilder();

        // 2. CriteriaQuery olustur (donus tipi: Product)
        CriteriaQuery<Product> cq = cb.createQuery(Product.class);

        // 3. Root tanimla (FROM kismi)
        Root<Product> root = cq.from(Product.class);

        // 4. WHERE kosulu ekle
        cq.select(root)
          .where(cb.greaterThan(root.get("price"), minPrice));

        // 5. Sorguyu calistir
        return em.createQuery(cq).getResultList();
    }
}

22.3 Ayni Sorgunun JPQL Karsiligi

Criteria API (8+ satir) JPQL (1 satir) 
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Product> cq = cb.createQuery(Product.class);
Root<Product> root = cq.from(Product.class);
cq.select(root)
  .where(cb.greaterThan(root.get("price"), minPrice));
return em.createQuery(cq).getResultList();
 
@Query("SELECT p FROM Product p WHERE p.price > :minPrice")
List<Product> findExpensive(@Param("minPrice") BigDecimal minPrice);

Kural: Eger sorgunuz statik ise (yani WHERE kosullari sabit), her zaman JPQL veya @Query kullanin. Criteria API sadece dinamik sorgular icin deger kazanir.

22.4 Eski Hibernate Criteria vs JPA Criteria

Tarihce

Eski Hibernate Criteria API (session.createCriteria()) daha basit bir API'ydi ancak JPA standardi degildi ve Hibernate 5+ ile deprecated oldu. JPA Criteria API standart olmasina ragmen daha karmasik ve okunaksizdir. Hibernate 6/7 ile yeni "Restrictions" yaklasimi bu sorunu hafifletmeye calismaktadir.

22.5 Dinamik Filtre Ornegi -- Criteria API'nin Parladigi Yer

Kullanici bir arama formunda farkli filtreleri secebilir: isim, kategori, fiyat araligi... Hangi filtrelerin gelecegi runtime'da belli olur. Bu senaryoda Criteria API cok degerlidir.

@Repository
public class ProductSearchRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> searchProducts(String name, String category,
                                         BigDecimal minPrice, BigDecimal maxPrice) {
        CriteriaBuilder cb = em.getCriteriaBuilder();
        CriteriaQuery<Product> cq = cb.createQuery(Product.class);
        Root<Product> root = cq.from(Product.class);

        // Dinamik olarak kosullari birlestir
        List<Predicate> predicates = new ArrayList<>();

        if (name != null && !name.isBlank()) {
            predicates.add(cb.like(cb.lower(root.get("name")),
                           "%" + name.toLowerCase() + "%"));
        }

        if (category != null && !category.isBlank()) {
            predicates.add(cb.equal(root.get("category"), category));
        }

        if (minPrice != null) {
            predicates.add(cb.greaterThanOrEqualTo(root.get("price"), minPrice));
        }

        if (maxPrice != null) {
            predicates.add(cb.lessThanOrEqualTo(root.get("price"), maxPrice));
        }

        // Tum kosullari AND ile birlestir
        cq.select(root)
          .where(cb.and(predicates.toArray(new Predicate[0])));

        return em.createQuery(cq).getResultList();
    }
}

Neden Criteria API Burada Parliyor?

Bu sorguyu JPQL ile yazmak icin string birlestirme ("SELECT ... " + (name != null ? "AND ..." : "")) gerekir ki bu hem okunaksiz hem de SQL injection risklidir. Criteria API, kosullari programatik olarak ve tip guvenli sekilde olusturmanizi saglar.

22.6 Spring Data JPA Specification Pattern

Spring Data JPA, Criteria API'yi saran daha temiz bir yaklasim sunar: Specification pattern.

// 1. Repository: JpaSpecificationExecutor'i extend et
public interface ProductRepository extends JpaRepository<Product, Long>,
                                           JpaSpecificationExecutor<Product> {
}

// 2. Specification sinifi olustur
public class ProductSpecifications {

    public static Specification<Product> hasName(String name) {
        return (root, query, cb) ->
            name == null ? null : cb.like(cb.lower(root.get("name")),
                                         "%" + name.toLowerCase() + "%");
    }

    public static Specification<Product> hasCategory(String category) {
        return (root, query, cb) ->
            category == null ? null : cb.equal(root.get("category"), category);
    }

    public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min != null && max != null) {
                return cb.between(root.get("price"), min, max);
            } else if (min != null) {
                return cb.greaterThanOrEqualTo(root.get("price"), min);
            } else if (max != null) {
                return cb.lessThanOrEqualTo(root.get("price"), max);
            }
            return null;
        };
    }
}

// 3. Servis katmaninda kullanimi
@Service
public class ProductService {

    @Autowired
    private ProductRepository repository;

    public List<Product> search(String name, String category,
                                BigDecimal minPrice, BigDecimal maxPrice) {
        Specification<Product> spec = Specification
            .where(ProductSpecifications.hasName(name))
            .and(ProductSpecifications.hasCategory(category))
            .and(ProductSpecifications.priceBetween(minPrice, maxPrice));

        return repository.findAll(spec);
    }
}

Ipucu: Specification pattern, Criteria API'nin gucunu Spring Data JPA'nin kolayligiyla birlestirir. Her filtre ayri bir metotta tanimlanir, yeniden kullanilabilir ve test edilebilir. Yeni bir filtre eklemek icin sadece yeni bir Specification metodu yazmak yeterlidir.

22.7 Hibernate 6/7: Yeni Restrictions Yaklasimi

JPQL + Dinamik WHERE Kosulu

Hibernate 6/7, Criteria API'nin karmasikligini azaltmak icin JPQL tabanlı dinamik sorgu olusturma imkani sunan yeni yaklasimlara yonelmektedir. Temel fikir: JPQL'in okunakliligini koruyarak WHERE kosullarini dinamik olarak eklemektir.

// Hibernate 6+ ile daha temiz dinamik sorgular
// HibernateCriteriaBuilder ek metodlar sunar
@Repository
public class ProductSearchRepository {

    @PersistenceContext
    private EntityManager em;

    public List<Product> search(String name, BigDecimal minPrice) {
        HibernateCriteriaBuilder cb = (HibernateCriteriaBuilder) em.getCriteriaBuilder();
        JpaCriteriaQuery<Product> cq = cb.createQuery(Product.class);
        JpaRoot<Product> root = cq.from(Product.class);

        List<Predicate> predicates = new ArrayList<>();

        if (name != null) {
            predicates.add(cb.ilike(root.get("name"), "%" + name + "%"));
            // ilike: case-insensitive LIKE (Hibernate 6+)
        }
        if (minPrice != null) {
            predicates.add(cb.ge(root.get("price"), minPrice));
        }

        cq.select(root).where(predicates.toArray(new Predicate[0]));
        return em.createQuery(cq).getResultList();
    }
}

22.8 Ne Zaman Hangisini Kullan?

Senaryo Onerilen Yaklasim Neden
Statik sorgu (WHERE kosullari sabit) JPQL / @Query Okunakli, basit, SQL benzeri
Dinamik filtreler (runtime'da belirlenir) Specification pattern Criteria API'yi sarar, yeniden kullanilabilir
Cok karmasik dinamik sorgular Criteria API (dogrudan) Tam kontrol, tip guvenligi
Karmasik raporlar, analitik sorgular jOOQ veya Native SQL SQL'in tam gucunu kullanir

Bilgi Kontrolu

1. Criteria API'nin en buyuk avantaji nedir?

✓ Dogru! Criteria API'nin en buyuk avantaji, WHERE kosullarini runtime'da dinamik olarak olusturabilmesidir. String birlestirme yerine tip guvenli nesneler kullanilir.
✗ Yanlis. Dogru cevap B: Criteria API, JPQL'den daha okunakli degildir ve performans farki yoktur. Gercek avantaji dinamik sorgu olusturma yetenegidir.

2. Statik bir sorgu icin (WHERE kosullari sabit) hangi yaklasim tercih edilmelidir?

✓ Dogru! Statik sorgularda JPQL veya @Query her zaman daha okunakli ve tercih edilir. Criteria API statik sorgularda gereksiz karmasiklik ekler.
✗ Yanlis. Dogru cevap A: Thorben'in de belirttigi gibi, statik sorgular icin JPQL her zaman daha okunaklidir. Criteria API sadece dinamik sorgularda deger kazanir.

3. Spring Data JPA Specification pattern'in avantaji nedir?

✓ Dogru! Specification pattern her filtreyi bagimsiz bir metot olarak tanimlar. Bu sayede filtreler yeniden kullanilabilir, test edilebilir ve .and() / .or() ile kolayca birlestirilebilir.
✗ Yanlis. Dogru cevap C: Specification pattern'in temel avantaji filtrelerin modular, yeniden kullanilabilir ve birlestirilebilir olmasidir. Performans farki yoktur, arka planda Criteria API kullanir.
